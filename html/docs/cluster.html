Cluster Bootstrap

A debz cluster is built around a single MASTER node that acts as WireGuard hub, Salt master, and Kubernetes control-plane endpoint. Nodes enroll via wg0, receive their identity, and are fully configured by Salt without any manual steps per node.

Architecture overview

The MASTER node (live USB or installed) brings up all four WireGuard planes with hub IPs (.1 on each network). Nodes boot into the debz live session with a DEBZ-SEED disk, install automatically, then request enrollment via wg0. The hub issues hub.env, and the node configures itself fully from there. 

# Cluster topology
MASTER (10.78.0.1) ←— Salt master, WG hub, K8s control plane
  ↑ wg1 (management)
  ├── kvm-02   10.78.0.2
  ├── kvm-03   10.78.0.3
  ├── storage-04  10.78.0.4
  └── storage-05  10.78.0.5

DEBZ-SEED disk

A DEBZ-SEED disk is a small FAT32-formatted disk or USB drive labeled DEBZ-SEED. It contains an answers.env file. When the debz live session boots and detects a disk with this label, debz-autoinstall.service reads the answers file and runs the installer non-interactively. 

# Create a DEBZ-SEED disk image (64 MB FAT32)
truncate -s 64M debz-seed.img
mkfs.fat -n DEBZ-SEED debz-seed.img

# Mount and write answers.env
mount debz-seed.img /mnt/seed
cat > /mnt/seed/answers.env <<EOF
DEBZ_HOSTNAME=kvm-02
DEBZ_PROFILE=kvm
DEBZ_DISK=/dev/sda
DEBZ_ZFS_TOPOLOGY=single
DEBZ_USERNAME=admin
DEBZ_PASSWORD=changeme
DEBZ_NET_METHOD=dhcp
DEBZ_TIMEZONE=UTC
DEBZ_LOCALE=en_US.UTF-8
EOF
umount /mnt/seed

# Write to a USB drive for physical hardware
dd if=debz-seed.img of=/dev/sdX bs=4M
Note The DEBZ-SEED disk is separate from the ISO boot drive. Boot from the debz ISO USB, and insert the DEBZ-SEED USB into a second port. The installer detects it automatically. For VMs, attach it as a second virtual disk image.

Enrollment flow

  1. 1
    Node boots from debz ISO. DEBZ-SEED detected. debz-autoinstall.service runs the installer with the answers file. System installs and reboots into the target OS.
  2. 2
    Installed node starts with wg0 pre-configured (private key generated at install, hub public key from answers.env). wg0 comes up on 10.77.0.0/16.
  3. 3
    debz-enroll.service sends an enrollment request to the hub on wg0: public keys for all four planes + proposed hostname.
  4. 4
    Hub’s debz-autobootstrap (30s loop) receives the request, assigns the next available node index, calls wg set on all four interfaces to add the peer, and issues a signed hub.env.
  5. 5
    Node receives hub.env, writes wg1/wg2/wg3 configs, brings all four interfaces up. Salt grains are written: role, node_index, wg1_ip, wg2_ip, wg3_ip, pubkeys.
  6. 6
    salt-minion starts, connects to 10.78.0.1 on wg1. Salt master auto-accepts the minion key (or requires manual approval if configured). Highstate runs and applies the node’s role.

Salt highstate flow

Each highstate run applies states in this order:

  1. base — common packages, NTP (chrony), nftables base rules, SSH hardening
  2. wireguard — regenerate wg configs from grains, ensure all planes are up
  3. role.<template> — template-specific packages, services, and configuration
  4. grains.update — re-write grains after role is applied
# Run highstate on a single minion from the master
salt 'kvm-02' state.highstate

# Run on all nodes
salt '*' state.highstate

# Check grain values on a node
salt 'kvm-02' grains.item debz:role debz:wg1_ip debz:node_index

# Accept all pending minion keys (auto-enrollment mode)
salt-key -A -y

Kubernetes cluster setup

Initialize the control plane on MASTER

# Run on the MASTER node (10.79.0.1 = wg2 address)
kubeadm init \
  --control-plane-endpoint=10.79.0.1 \
  --apiserver-advertise-address=10.79.0.1 \
  --pod-network-cidr=10.244.0.0/16 \
  --skip-phases=addon/kube-proxy

# Configure kubectl
mkdir -p ~/.kube
cp /etc/kubernetes/admin.conf ~/.kube/config

# Install Cilium (kube-proxy replacement, WireGuard backend on wg2)
helm install cilium cilium/cilium \
  --namespace kube-system \
  --set kubeProxyReplacement=true \
  --set k8sServiceHost=10.79.0.1 \
  --set k8sServicePort=6443

Join worker nodes

# On each worker node, run debz-k8s-join
# This fetches the join command from Salt mine and joins with the correct wg2 IP
debz-k8s-join

# Equivalent manual steps:
# 1. Get join command from master
kubeadm token create --print-join-command

# 2. On the worker, specifying wg2 IP as the node IP
kubeadm join 10.79.0.1:6443 --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --node-ip=$(debz-wg-ip wg2)

Verify cluster nodes

# From the MASTER node
kubectl get nodes -o wide

# All node IPs should be wg2 addresses (10.79.0.x)
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.addresses[0].address}{"\n"}{end}'

Example: 5-node cluster (1 master, 2 KVM, 2 storage)

# 1. Build golden images
./deploy.sh golden TEMPLATE=kvm
./deploy.sh golden TEMPLATE=storage

# 2. Stamp clones
./deploy.sh stamp TEMPLATE=kvm   STAMP_COUNT=2 STAMP_BASE_HOSTNAME=kvm     STAMP_BASE_INDEX=2
./deploy.sh stamp TEMPLATE=storage STAMP_COUNT=2 STAMP_BASE_HOSTNAME=storage STAMP_BASE_INDEX=4

# 3. Spawn all clones on Proxmox
./deploy.sh spawn TEMPLATE=kvm     PROXMOX_HOST=proxmox.debz.local
./deploy.sh spawn TEMPLATE=storage PROXMOX_HOST=proxmox.debz.local

# 4. Start MASTER (live USB or installed), bring up hub
# Boot MASTER node, wg0-wg3 come up automatically

# 5. Boot the clones — each enrolls automatically via wg0
# Watch enrollment on master
journalctl -fu debz-autobootstrap.service

# 6. Accept Salt keys and run highstate
salt-key -A -y
salt '*' state.highstate

# 7. Initialize Kubernetes
debz-k8s-join   # run on each KVM node
kubectl get nodes  # verify from MASTER
Autobootstrap log All enrollment activity is logged to /var/log/debz/autobootstrap.log on the MASTER node. Tail this file to watch nodes enroll in real time.
