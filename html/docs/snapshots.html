Snapshot System

debz runs two automatic snapshot policies: APT hooks for the root filesystem, and a 15-minute timer for /srv. Both use count-based retention — no time-based expiry.

Two independent snapshot policies

PolicyTriggerDatasetNamingRetention
APT hook Before every apt/dpkg operation rpool/ROOT/<hostname> @apt-YYYYMMDD-HHMMSS Keep last 10
/srv timer Every 15 minutes (systemd timer) rpool/srv @srv-YYYYMMDD-HHMMSS Keep last 4

APT snapshot hooks

Debian’s APT and dpkg both support pre/post operation hooks. debz installs hooks in /etc/apt/apt.conf.d/ and /etc/dpkg/dpkg.cfg.d/ that snapshot rpool/ROOT/<hostname> before package operations begin.

The snapshot is taken before any package is unpacked or configured. If a package installation breaks the system, you have a clean snapshot from immediately before the operation.

# Snapshot created automatically before apt install
apt install nginx

[debz-snapshot] Snapshotting rpool/ROOT/prod-node-01 ...
[debz-snapshot] Created: rpool/ROOT/prod-node-01@apt-20260315-142201
[debz-snapshot] Pruning: kept last 10, removed 0
Reading package lists... Done
...

What triggers the APT hook

  • apt install
  • apt remove / apt purge
  • apt upgrade / apt dist-upgrade
  • dpkg -i / dpkg --configure
  • Any package manager that invokes dpkg underneath

The hook does not trigger on apt update (no package changes) or apt-cache queries.

/srv timer

A systemd timer snapshots rpool/srv every 15 minutes. This is intended for workloads that store mutable service data in /srv — databases, object stores, git repositories, content directories.

The timer is recursive: child datasets under rpool/srv are snapshotted with a single atomic zfs snapshot -r call, ensuring consistency across all /srv sub-trees at the same timestamp. 

# Check timer status
systemctl status debz-snapshot-srv.timer

● debz-snapshot-srv.timer - debz /srv snapshot timer
     Loaded: loaded (/usr/lib/systemd/system/debz-snapshot-srv.timer; enabled)
     Active: active (waiting)
    Trigger: 2026-03-15 14:45:00 UTC (in 8min 33s)
   Triggers: ● debz-snapshot-srv.service

# Run a manual snapshot right now
systemctl start debz-snapshot-srv.service

Count-based retention

debz uses count-based retention only — there is no time-based expiry. After each snapshot, the pruning logic lists all snapshots matching the policy prefix on the dataset, sorts them oldest-first, and destroys any beyond the retention count.

This means:

  • If APT runs 15 times in one day, you have the last 10 of those snapshots — regardless of age
  • If /srv has not changed in 3 days, you still have the last 4 timer snapshots
  • Storage usage is predictable and bounded — old snapshots do not accumulate
Note ZFS snapshots are copy-on-write references. A snapshot of a 10 GB dataset does not immediately consume 10 GB of additional space — it only holds the delta between the snapshot point and the current state. If you haven’t changed many files, snapshots are very cheap.

Rolling back to a snapshot

# Roll back the root filesystem to an APT snapshot
# WARNING: discards all changes since that snapshot was taken
zfs rollback rpool/ROOT/prod-node-01@apt-20260315-142201

# Roll back /srv to a point in time
zfs rollback rpool/srv@srv-20260315-140001
Warning zfs rollback destroys all snapshots newer than the target snapshot by default. Use zfs rollback -r to also destroy dependent clones and newer snapshots in one operation. This cannot be undone — take note of what you are losing before running rollback.

For the root filesystem, prefer using debz-be activate <name> and rebooting over a direct zfs rollback — the BE manager handles the logistics of making the snapshot bootable without destroying intermediate snapshots.

Restoring individual files from a snapshot

ZFS snapshots are accessible as a hidden .zfs/snapshot/ directory at the dataset root:

# Browse snapshot contents without mounting
ls /srv/.zfs/snapshot/

srv-20260315-140001  srv-20260315-141501  srv-20260315-143001  srv-20260315-144501

# Restore a single file from 2 snapshots ago
cp /srv/.zfs/snapshot/srv-20260315-141501/myapp/data.db /srv/myapp/data.db

# Restore a directory tree
rsync -a /srv/.zfs/snapshot/srv-20260315-140001/myapp/ /srv/myapp/

Manual snapshots

# Take a manual snapshot before a risky operation
zfs snapshot rpool/srv@pre-migration-20260315

# Or use debz-be for the root dataset
debz-be create pre-config-change

# Manual snapshots are NOT subject to automatic pruning
# They must be destroyed manually when no longer needed
zfs destroy rpool/srv@pre-migration-20260315
Tip Manual snapshots use a different naming prefix from automatic ones (@apt- and @srv-). The pruning logic only targets snapshots with these prefixes, so manual snapshots are never automatically deleted — you own their lifecycle.
