Extending debz

debz is a substrate, not a straitjacket. Every layer from the installer payload to the Kubernetes CNI is designed to be extended without forking. This page covers the full extension surface: the DEBZ-SEED payload structure, postinstall.sh hooks, custom apt mirrors, Ansible, Helm, Cilium, node discovery, and drop-in Debian integration.

1. The DEBZ-SEED disk — full payload reference

Most documentation describes answers.env as the sole content of a DEBZ-SEED disk. That is just the minimum. The installer reads and stages a full payload directory tree. Everything beyond answers.env is optional but processed automatically when present. 

# Complete DEBZ-SEED directory structure
DEBZ-SEED (FAT32, labeled volume)
├── answers.env          # required — install answers (profile, disk, hostname, user…)
├── postinstall.sh       # optional — runs in installed chroot on firstboot
├── repos/               # optional — copied to /etc/apt/sources.list.d/
│   ├── mycompany.list
│   └── mycompany.gpg
├── salt/                # optional — copied to /srv/salt/
│   ├── top.sls
│   └── roles/
├── ansible/             # optional — copied to /etc/debz/ansible/
│   ├── site.yml
│   └── inventory/
└── darksite/            # optional — additional apt package pool merged with ISO mirror
    └── pool/
        └── main/

How the installer stages the payload

When debz-autoinstall.service detects a DEBZ-SEED labeled disk, it mounts it and processes each directory in order after the core install completes:

  1. repos/ — all .list and .gpg files are copied into /etc/apt/sources.list.d/ in the installed chroot, so they are available from the very first apt-get run on the target system.
  2. salt/ — the entire directory is rsync’d into /srv/salt/ on the target. If a Salt master is already configured, the states are immediately available on the next highstate.
  3. ansible/ — rsync’d into /etc/debz/ansible/. Your postinstall.sh can then run ansible-playbook against this content from within the chroot.
  4. darksite/pool/ — merged into the existing darksite apt pool so apt-get can install packages from your private mirror without any extra repo configuration.
  5. postinstall.sh — staged to /usr/lib/debz/firstboot/postinstall.sh on the target and executed by debz-firstboot.service on the first boot of the installed system.
Order matters Repos are staged before postinstall.sh runs, so you can apt-get install packages from your private mirror inside the hook without any extra setup.

Creating a DEBZ-SEED disk with a full payload

# Create a 256 MB DEBZ-SEED image
truncate -s 256M debz-seed.img
mkfs.fat -n DEBZ-SEED debz-seed.img
mount debz-seed.img /mnt/seed

# Required
cp answers.env /mnt/seed/

# Optional: apt repo + GPG key
mkdir /mnt/seed/repos
cp mycompany.list mycompany.gpg /mnt/seed/repos/

# Optional: Ansible playbooks
mkdir -p /mnt/seed/ansible/inventory
cp site.yml /mnt/seed/ansible/
cp hosts.yaml /mnt/seed/ansible/inventory/

# Optional: firstboot hook
cp postinstall.sh /mnt/seed/

umount /mnt/seed

# Write to USB for physical hardware
dd if=debz-seed.img of=/dev/sdX bs=4M status=progress oflag=sync

2. postinstall.sh

postinstall.sh is an arbitrary shell script that runs inside the installed system’s chroot on firstboot, after Salt highstate has applied the node’s role. It has full access to the installed system’s filesystem, apt-get, and the network (if available). Staged repos from the DEBZ-SEED repos/ directory are already active.

Execution environment

PropertyValue
Runs asroot inside the installed system chroot
Invoked bydebz-firstboot.service (oneshot, After=network-online.target)
Working directory/ (root of installed system)
Seed payload at/var/lib/debz/seed-payload/
Log output/var/log/debz/firstboot.log
Exit codeNon-zero marks firstboot as failed; service reports failure via journald

Idempotency debz-firstboot.service is a oneshot that runs exactly once. If you need to re-run your hook during testing, use systemctl reset-failed debz-firstboot.service && systemctl start debz-firstboot.service.

Example 1 — Add a private apt repository

#!/usr/bin/env bash
# postinstall.sh — add company apt repo and install agent
set -euo pipefail

curl -fsSL https://apt.mycompany.com/key.gpg \
  | gpg --dearmor > /usr/share/keyrings/mycompany.gpg

echo "deb [signed-by=/usr/share/keyrings/mycompany.gpg] \
  https://apt.mycompany.com/debian trixie main" \
  > /etc/apt/sources.list.d/mycompany.list

apt-get update -qq
apt-get install -y mycompany-agent

systemctl enable --now mycompany-agent

Example 2 — Run an Ansible playbook

#!/usr/bin/env bash
set -euo pipefail

# Ansible + playbooks are staged from the DEBZ-SEED ansible/ directory
cd /etc/debz/ansible
ansible-playbook -i inventory/hosts site.yml --connection=local

Example 3 — Clone Salt states from a company repo

#!/usr/bin/env bash
set -euo pipefail

git clone https://git.mycompany.com/salt-states /srv/salt/mycompany

# Extend the Salt master file-roots with the new path
echo "- /srv/salt/mycompany" >> /etc/salt/master.d/file-roots.conf
systemctl reload salt-master 2>/dev/null || true

Example 4 — Full ETL pipeline node setup

#!/usr/bin/env bash
# postinstall.sh — ETL pipeline node
set -euo pipefail

apt-get install -y apache-spark python3-pyspark kafka-clients

# Copy pipeline configs from the staged seed payload
cp -r /var/lib/debz/seed-payload/pipeline-configs /etc/etl/

# Write a node-specific config using injected variables
cat > /etc/etl/node.conf <<EOF
ETL_NODE_ID=${DEBZ_NODE_INDEX}
ETL_NODE_IP=${DEBZ_WG1_IP}
ETL_HOSTNAME=${DEBZ_HOSTNAME}
EOF

systemctl enable --now etl-worker

3. Custom darksite repositories

The debz ISO ships with a full APT mirror baked in under /root/darksite/ on the live system. This mirror is what the installer uses when no internet is available. You can augment it with your own packages via two approaches.

a) Runtime: DEBZ-SEED darksite/ directory

Place .deb files under darksite/pool/main/ on the DEBZ-SEED disk. The installer merges this directory into the live darksite pool before running apt-get, so your packages are available during the install and in the installed system via the on-disk mirror. This is per-node and requires no ISO rebuild. 

# Add private debs to the seed disk
mkdir -p /mnt/seed/darksite/pool/main
cp mypackage_1.0_amd64.deb /mnt/seed/darksite/pool/main/

# The installer calls dpkg-scanpackages and regenerates Packages.gz automatically

b) Build-time: bake packages into every ISO

Add packages to the live-build darksite tree before running ./deploy.sh build. All debs placed here are included in the ISO’s embedded mirror and available on every node deployed from that ISO, with no seed disk required. 

# Place debs into the build tree
cp mypackage_1.0_amd64.deb \
  live-build/config/includes.chroot/root/darksite/apt/pool/main/

# Rebuild the ISO
./deploy.sh build

# All nodes installed from this ISO can install mypackage without internet
apt-get install mypackage
Which approach to use? Use the DEBZ-SEED darksite/ directory when your private packages change frequently or differ per-site. Use the build-time approach when every ISO deployment needs the same base packages — reduces per-node seed disk size and simplifies fleet management.

4. Ansible integration

Ansible, ansible-lint, and ansible-core are pre-installed on the live ISO via cloud-tools.list.chroot. debz does not replace Ansible — it provides a solid, rollback-capable, ZFS-backed base that Ansible can target like any other Debian system.

Bootstrapping Ansible from postinstall.sh

# postinstall.sh — pull playbooks from Git and converge
set -euo pipefail

git clone https://git.mycompany.com/ansible /etc/debz/ansible

# Install collections offline from the seed payload
ansible-galaxy collection install \
  /var/lib/debz/seed-payload/collections/community.general.tar.gz

# Converge against localhost using node variables
ansible-playbook \
  -i /etc/debz/ansible/inventory/hosts \
  /etc/debz/ansible/site.yml \
  --connection=local \
  --extra-vars "debz_profile=${DEBZ_PROFILE} debz_index=${DEBZ_NODE_INDEX}"

Dynamic inventory using Salt grains

If you already use Salt for configuration management, you can drive Ansible with a dynamic inventory script that queries Salt grains. This gives you a single source of truth for host metadata across both tools. 

#!/usr/bin/env python3
# salt-inventory.py — dynamic Ansible inventory from Salt grains
import json, subprocess

result = subprocess.run(
    ['salt', '*', 'grains.items', '--out=json'],
    capture_output=True, text=True
)
grains = json.loads(result.stdout)

inventory = {'_meta': {'hostvars': {}}, 'debz': {'hosts': []}}
for host, g in grains.items():
    inventory['debz']['hosts'].append(host)
    inventory['_meta']['hostvars'][host] = {
        'ansible_host': g.get('debz:wg1_ip'),
        'debz_role': g.get('debz:role'),
        'debz_index': g.get('debz:node_index'),
    }

print(json.dumps(inventory, indent=2))
# Run a playbook against all debz nodes via Salt-sourced inventory
ansible-playbook -i salt-inventory.py site.yml

Targeting debz node roles in playbooks

# site.yml — role-targeted playbook for debz nodes
- name: Base configuration
  hosts: debz
  become: true
  tasks:
    - name: Ensure chrony is running
      ansible.builtin.service:
        name: chrony
        state: started
        enabled: true

- name: KVM hypervisor tuning
  hosts: debz
  become: true
  when: debz_role == 'kvm'
  tasks:
    - name: Set huge pages
      ansible.posix.sysctl:
        name: vm.nr_hugepages
        value: '1024'

5. Helm + Kubernetes

Helm is pre-installed on all debz nodes via 0075-cloud-tools.hook.chroot. kubectl, kubeadm, and kubelet are also pre-installed. After debz-k8s-join completes, the cluster behaves exactly like any kubeadm cluster — deploy Helm charts exactly as you would on any Kubernetes distribution.

Deploy common platform charts

# Add chart repos
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo add jetstack     https://charts.jetstack.io
helm repo add argo         https://argoproj.github.io/argo-helm
helm repo update

# Install nginx ingress controller
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx --create-namespace

# Install cert-manager with CRDs
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager --create-namespace \
  --set installCRDs=true

# Install ArgoCD
helm install argocd argo/argo-cd \
  --namespace argocd --create-namespace

All traffic over wg2

Every debz Kubernetes node binds its kubelet to its wg2 address (10.79.0.x). Cilium uses the same plane as its underlay. When deploying services, pods will communicate exclusively over the encrypted WireGuard mesh — no unencrypted lateral traffic is possible. 

# Verify node IPs are wg2 addresses
kubectl get nodes -o wide
# NAME        STATUS   ROLES    AGE   VERSION   INTERNAL-IP
# master-01   Ready    control  12m   v1.30.0   10.79.0.1
# kvm-02      Ready    worker   8m    v1.30.0   10.79.0.2
# kvm-03      Ready    worker   6m    v1.30.0   10.79.0.3

6. Cilium + Hubble

Cilium is installed as the Kubernetes CNI by debz-autobootstrap with kubeProxyReplacement=true. It operates on top of the wg2 plane (10.79.x.x), meaning all east-west Kubernetes traffic is double-encrypted: once by WireGuard at the kernel level, and once by Cilium’s eBPF policies.

Hubble observability

Hubble is Cilium’s observability layer. The Hubble CLI is pre-installed on all debz nodes. It gives you real-time visibility into every L3/L4/L7 flow in your cluster without any sidecars, agents, or service mesh overhead. 

# Check Hubble status
hubble status

# Observe live flows — all namespaces
hubble observe --follow

# Filter: only flows to port 443
hubble observe --follow --port 443

# Filter: flows from a specific pod
hubble observe --from-pod default/my-app --follow

# Access Hubble UI in a browser via port-forward
kubectl port-forward -n kube-system svc/hubble-ui 12000:80
# Open: http://localhost:12000

eBPF tooling on all nodes

bpftool, bpftrace, and bcc-tools are pre-installed on every debz node. Attach kernel probes, trace syscalls, inspect maps, and dump loaded programs without rebooting or installing anything. 

# List all loaded eBPF programs (including Cilium's)
bpftool prog list

# Trace TCP retransmits in real time
bpftrace -e 'kprobe:tcp_retransmit_skb { printf("%s\n", comm); }'

# Count syscalls by process name using bcc
syscount -p $(pgrep kubelet)

7. Node discovery

debz uses a two-layer discovery system. Enrolled nodes are tracked in Salt mine. Unenrolled live nodes broadcast their presence over mDNS. Both sources are visible in the Web UI.

a) Enrolled nodes — Salt mine

Once a node has enrolled and highstate has run, its identity, role, and WireGuard IPs are stored as Salt grains and available via Salt mine queries from anywhere in the cluster. 

# List all enrolled nodes with role, node_index, and WG IPs
salt '*' grains.items

# Filter: only KVM nodes
salt -G 'debz:role:kvm' grains.item debz:wg1_ip debz:node_index

# Check all nodes are reachable
salt '*' test.ping

b) Unenrolled live nodes — Avahi mDNS

When a debz live session boots, debz-announce.service broadcasts a _debz._tcp mDNS service record on the local network. The MASTER’s Web UI discovers these records and presents unenrolled nodes as enrollment targets in the Spawn panel. 

# Browse for live debz nodes on the LAN
avahi-browse -rt _debz._tcp

# Output format:
# = eth0 IPv4 debz-live-kvm-03  _debz._tcp  local
#   hostname = [kvm-03.local]
#   address = [10.0.1.43]
#   port = [8080]
#   txt = [profile=server version=1]
Web UI enrollment The Web UI Spawn panel shows discovered mDNS nodes as “Discovered — click to enroll” tiles. Clicking one opens the enrollment wizard pre-filled with the detected IP and hostname, requiring only template selection and confirmation to proceed.

c) Manual enrollment

If a node is on a different broadcast domain or mDNS is unavailable, you can enroll it manually by entering its IP address in the Spawn panel’s manual entry field. The MASTER will attempt to reach it over wg0 and initiate enrollment. 

# Manual enrollment from the command line on MASTER
debz-enroll-peer --ip 10.0.2.50 --hostname storage-06 --role storage

8. Integrating into an existing environment

debz is Debian 13 (trixie) underneath — the same dpkg, the same APT, the same package namespace, the same syscall ABI. If your team already uses Debian, replacing your current install ISO with debz is the fastest path to ZFS boot environments, WireGuard mesh, and cluster tooling without touching a single existing workflow.

What you keep

Existing toolingWorks on debz?Notes
Ansible playbooks ✓ Unchanged Target debz nodes exactly like any Debian host
Terraform / Pulumi ✓ Unchanged Provision via Proxmox API, AWS EC2, GCP, Azure ARM directly
Chef / Puppet ✓ Unchanged Install the agent in postinstall.sh or from apt
Docker / Podman containers ✓ Unchanged Both are available; images and compose files work as-is
Kubernetes workloads ✓ Unchanged Same kubeadm join, same kubectl, same manifest YAML
Existing Debian apt repos ✓ Unchanged Add to /etc/apt/sources.list.d/ as normal
systemd units and timers ✓ Unchanged Standard systemd; debz adds units with debz- prefix only
CI/CD pipelines ✓ Unchanged SSH access works immediately; GitLab runner, Jenkins agent, etc.

What you gain

  • ZFS on root — every package operation is preceded by a snapshot. Roll back any bad upgrade in 30 seconds.
  • Boot environments — ZFSBootMenu lets you boot into any prior OS snapshot from the bootloader menu.
  • Encrypted WireGuard mesh — four planes with automatic peer management. Lateral traffic is always encrypted.
  • Golden image pipeline — stamp 50 CoW clones of any node in minutes. Each clone is ~1 MB until written.
  • Air-gap capability — the embedded apt mirror means the ISO and all installed packages are available with no internet.

No lock-in: vendor APIs, not wrappers

debz uses provider APIs directly — Proxmox REST API, AWS EC2, GCP Compute, Azure ARM. There are no debz-specific wrappers between your code and the cloud. deploy.sh is a thin orchestrator that calls standard tools (qemu-img, virsh, pvesh, aws ec2). If you outgrow it, replace it.

9. What can you build?

debz is a general-purpose host OS. The ZFS + WireGuard + Salt + Kubernetes stack is not prescriptive — it’s a set of primitives you combine for your workload. Here are nine reference architectures.

🌐

Landing sites + web apps

Run nginx or Caddy on a KVM or STORAGE node. ZFS snapshots protect /srv/www every 15 minutes. Replicate over wg3 to an offsite node for instant failover. Deploy via Ansible or Helm. Serve from bare metal or a Proxmox VM with sub-millisecond disk I/O.

📸

Remote desktop (VDI)

Deploy the VDI template: mediamtx ingests a wf-recorder SRT stream and re-publishes it as HLS and WebRTC. Any browser connects to port :8889 — zero client install required. Optional NVIDIA nvenc for GPU-accelerated encode. Wayland desktop fully headless.

⚡

MicroVMs + serverless

Firecracker is pre-installed on KVM nodes. MicroVMs boot in under 125 ms. Stamp 50 CoW clones of a golden Firecracker rootfs in minutes. Combine with the golden image pipeline to achieve serverless-like cold-start times on bare metal you own.

🗄️

Storage cluster

STORAGE template nodes export ZFS pools over NFS, iSCSI (tgt), and Samba via wg3. zfs send/recv replicates datasets to TrueNAS, another debz node, or a cloud VM. 15-minute snapshots of /srv give a 15-minute RPO with zero additional tooling.

📈

ETL pipeline

KVM nodes running Apache Spark and Kafka with data on ZFS. The 15-minute /srv snapshot schedule protects pipeline state. Use postinstall.sh to install pipeline-specific packages and configs from the DEBZ-SEED payload. Node index and wg1 IP are available as environment variables for per-node configuration.

✨

Kubernetes cluster

MASTER + KVM worker nodes with Cilium CNI, Hubble observability, and Helm. All east-west traffic over wg2. debz-k8s-join fetches the join command from Salt mine automatically. Workers join and are fully operational in under 90 seconds from first boot.

✈️

Air-gapped data centre

Build the ISO once with your private packages baked into the darksite mirror. Deploy 100 nodes from USB or PXE with zero internet access at any stage — install, firstboot, highstate, and package upgrades all run from the embedded mirror. DEBZ-SEED darksite directories add site-specific packages per rack.

🚀

Dev / CI lab

Build one golden image per test environment definition. debz-stamp-identity CoW-clones a fresh environment in 30 seconds with a unique hostname, machine-id, and WireGuard identity. Delete the clone after the CI run. Storage cost is ~1 MB per environment until divergence. Combine with Firecracker for sub-second environment creation.

🏠

Hybrid cloud

The same Salt states, Ansible playbooks, and Helm charts target on-prem KVM nodes and AWS EC2, GCP Compute, or Azure ARM VMs simultaneously. WireGuard wg1 extends across cloud providers — Salt master on-prem reaches cloud minions over the same management plane. No VPC peering, no transit gateway, no lock-in.
