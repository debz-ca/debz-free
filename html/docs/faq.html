Frequently Asked Questions

Philosophy, use cases, architecture decisions, and the questions people ask before they burn the ISO.

Philosophy

What is debz, really?

debz is a Debian 13 host OS distribution built around a single idea: the infrastructure layer should be invisible and indestructible. ZFS on root, four WireGuard planes, Salt configuration management, and a browser-based control plane are baked into a single ISO. You burn it, boot it, and your machine is already a cluster node — no post-install configuration, no playbook bootstrapping, no setup guide to follow.

It is not a Kubernetes distribution. It is not a hypervisor. It is not an opinionated application platform. It is the thin layer of infrastructure underneath whatever you are building — made reliable, reproducible, and recoverable by default.

Why build this instead of using an existing distro?

Every existing option makes you choose: you get a good OS or a good cluster substrate, not both. Talos is locked down to the point of uselessness for mixed workloads. Ubuntu Server gives you nothing out of the box and punishes you with snap. Proxmox VE is a hypervisor appliance that fights you when you need a general-purpose host. NixOS requires you to learn a new language before you can do anything.

debz is just Debian — with the boring parts (ZFS setup, WireGuard key generation, Salt bootstrap, snapshot policy) done for you, reproducibly, in ~90 seconds. Every apt package, every Docker image, every Kubernetes manifest you already use continues to work. Nothing is replaced. The floor is just higher.

Is this opinionated? Will it force a specific workflow on me?

The storage layout is fixed: ZFS on root, deterministic dataset hierarchy, ZFSBootMenu. That is non-negotiable and intentional — it is what makes boot environments and reliable rollbacks possible.

Everything else is a primitive you use or ignore. WireGuard planes are configured but you control what services bind to which plane. Salt is installed but you bring your own states. Kubernetes bootstrapping is provided but your CNI, your ingress, your storage CSI are your choice. The golden image pipeline is a tool, not a mandate.

If you want to run your Ansible playbooks unchanged on top of debz, you can. If you want to ignore Salt entirely and use Puppet, you can. The OS gets out of your way.

What does "100% bash" mean?

The installer, the firstboot scripts, the snapshot system, the boot environment manager, the WireGuard enrollment, the darksite builder — all of it is plain bash. No Go binary to cross-compile. No Python virtualenv to manage. No Ruby gems. No compiled operator to debug at 2am when a node won't boot.

Bash scripts are readable by any Linux admin without prior debz knowledge. You can read every line of what ran on your machine. You can modify it. You can vendor it. The Web UI server is the only exception — it is a single Python3 file with no dependencies beyond the standard library.

Who is debz for?

debz is for people who run real infrastructure and are tired of the accidental complexity that accumulates around it. Specifically:

  • Homelab operators who want a proper, reproducible cluster without cloud pricing.
  • Small infrastructure teams who need a reliable base for bare-metal Kubernetes, KVM hypervisors, or mixed workloads.
  • Security-conscious environments that need air-gap installs, encrypted ZFS, Secure Boot, and no surprise outbound connections.
  • Anyone who has been burned by a botched apt upgrade with no rollback path, by an un-reproducible node, or by a cloud bill that arrived without warning.

It is probably not for you if you are happy in a managed cloud with managed Kubernetes and never need to think about the host OS.

Why ZFS

Why ZFS instead of ext4/btrfs/LVM?

Three things no other Linux filesystem gives you together: atomic snapshots, boot environments, and end-to-end checksumming.

Atomic snapshots mean a snapshot before every apt upgrade is zero-cost and takes under a second. If the upgrade breaks something, you roll back the entire OS in 30 seconds without a rescue USB. On ext4 or btrfs you get neither the atomicity nor the boot-environment integration to make rollback seamless.

Boot environments mean each OS state is a separate dataset that ZFSBootMenu can present as a boot option. You can have a pre-upgrade, a post-upgrade, and a known-good state all on the same disk, switchable at boot without reinstalling anything.

Checksumming means silent data corruption — bit rot, bad RAM, flaky storage controllers — is detected and (with redundancy) automatically repaired. On a filesystem without checksums you find out about corruption when an application crashes, not before.

Is ZFS stable on Linux? I've heard it's a licensing problem.

OpenZFS on Linux has been production-stable for years. It ships in Ubuntu, Proxmox, TrueNAS, and dozens of enterprise distributions. The CDDL/GPL licensing incompatibility means it cannot be merged into the mainline kernel, but it has shipped as a separate out-of-tree module in mainstream Linux distros since 2016 without issue.

Debian ships OpenZFS packages in the contrib component. debz pins the version and holds it during upgrades so you never accidentally land on an untested kernel/ZFS combination.

Does ZFS use a lot of RAM?

The ZFS ARC (Adaptive Replacement Cache) is dynamic — it uses free RAM and shrinks under memory pressure. On a 4 GB machine it will use ~1 GB. On a 64 GB machine it might use 20 GB. This is a feature, not a bug: unused RAM is wasted RAM, and reads hitting the ARC are effectively free.

If you are running a memory-constrained system (Raspberry Pi, small VM), you can cap the ARC: echo "options zfs zfs_arc_max=536870912" > /etc/modprobe.d/zfs.conf (512 MB in that example).

The minimum practical RAM for a debz node is 2 GB. 4 GB+ is comfortable for a headless server. The desktop profile wants 8 GB or more.

What happens if the ZFS pool gets corrupted or a disk dies?

A single-disk pool (the default) has no redundancy — if the disk dies, data is gone, same as any single-disk setup. What debz gives you over a plain ext4 install is early warning and snapshot-based recovery: ZFS detects and logs checksum errors before they become silent corruption, and your most recent snapshots are on the same pool.

For redundancy, use a mirror or RAIDZ topology at install time. debz-install-target supports DEBZ_ZFS_TOPOLOGY=mirror with two disks. On a mirror, a complete disk failure is handled transparently — resilver the replacement and you are back to full redundancy.

The debz-recovery tool handles pool import and chroot repair from the live ISO in cases where the installed system won't boot.

How do boot environments work in practice?

Each boot environment is a ZFS snapshot or clone of rpool/ROOT/default with its own mountpoint. ZFSBootMenu presents them at boot as a menu. You pick the one you want and the system boots into it — full OS state, all of /, exactly as it was when the snapshot was taken.

debz creates a boot environment automatically before every APT/dpkg operation via an APT hook. So every apt upgrade, every package install, leaves a recoverable pre-state. The policy keeps the last 10 — old ones are pruned automatically.

To manage BEs manually: debz-be list, debz-be create my-label, debz-be activate my-label, debz-be rollback.

WireGuard

Why four WireGuard planes? Why not just one VPN?

Separation of concerns. Traffic classes have different security, performance, and failure domain requirements that you cannot satisfy with a single tunnel:

  • wg0 (enrollment) — untrusted new nodes connect here for bootstrap only. Isolated so a compromised new node cannot reach management traffic.
  • wg1 (management) — Salt master, SSH, internal APIs. Admin access only. Even if your application plane is compromised, management remains isolated.
  • wg2 (Kubernetes) — pod-to-pod and API server traffic. CNI overlay rides on top. Isolated so Kubernetes networking problems don't affect storage or management.
  • wg3 (storage) — ZFS replication, NFS, iSCSI. High throughput, low jitter. Isolated so a storage flood doesn't starve the management plane.

In practice this also makes firewall rules simple and auditable: each plane has its own subnet, port, and nftables chain. You can block or shape each class independently.

How do new nodes get their WireGuard keys?

Each node generates its own keypairs at first boot. The public keys are registered with the master during enrollment. The master adds each node as a peer on all four interfaces without restarting any WireGuard interface — wg set adds peers live.

Node IPs are deterministic by design: a node at index 5 always gets 10.78.100.5/32 on wg1, 10.79.100.5/32 on wg2, etc. No DHCP, no DNS, no coordination required to know the IP of any node from its index.

Can debz nodes connect to non-debz peers on WireGuard?

Yes. WireGuard is standard WireGuard. Any device with a WireGuard client — a phone, a laptop, an AWS VPC, a cloud VM running plain Ubuntu — can be added as a peer on any plane. The four-plane structure is a convention debz enforces for its own nodes; nothing prevents you from adding arbitrary peers manually with wg set or wg-add-peer.

What happens to WireGuard connectivity if the master node reboots?

WireGuard interfaces come up via wg-quick@wgN.service on boot, which reads the persisted /etc/wireguard/wgN.conf. All previously enrolled peers are in that file. Connectivity resumes as soon as the interfaces are up — no re-enrollment needed. The master's reboot is transparent to already-enrolled nodes (their keepalive packets will resume connecting once the master is back).

Salt & Kubernetes

Do I have to use Salt?

No. Salt is installed on every node and the master is pre-configured, but you are not required to use it. If you use Ansible, bring your playbooks — debz nodes are standard Debian hosts reachable over SSH on wg1. If you use Puppet, Chef, or nothing at all, that's fine too.

Salt is there because the cluster bootstrap, node registration, and Kubernetes join workflow use it internally. Once your cluster is up, how you manage configuration on top is entirely your call.

Does debz manage my Kubernetes workloads with Salt?

No. Salt is used by debz for infrastructure convergence only — package installs, WireGuard key distribution, node role assignment, and Kubernetes join commands. It does not manage your deployments, your Helm releases, your ingress, or your application config. Those are yours.

The boundary is: Salt manages the node. Kubernetes manages the workload. They do not overlap.

Which Kubernetes distribution does debz use?

Vanilla upstream Kubernetes — kubeadm init, containerd, your choice of CNI. debz does not ship a custom Kubernetes distribution, a custom scheduler, or any Kubernetes-level modifications. You get the same kubeadm-bootstrapped cluster you would configure by hand, just done automatically and repeatably.

The default CNI is Flannel. Calico, Cilium, and others work fine — swap the CNI manifest before running kubeadm init.

Can I run a single-node "cluster"?

Yes. Install the etcd template on one machine and it will bootstrap as both control plane and worker (with the control plane taint removed). This is useful for development, CI runners, or edge deployments where you want Kubernetes scheduling without the overhead of three etcd nodes.

For production, three etcd nodes and separate workers is the recommended minimum for HA.

What version of Salt does debz use?

Salt 3006 LTS, installed from the official SaltProject (Broadcom) APT repository and pinned at that major version. The packages are fetched at ISO build time and baked into the darksite APT pool, so installs are fully offline — no Broadcom repo access required at node provision time.

Use Cases

I have a Proxmox cluster. Where does debz fit?

Two ways. Either replace your Proxmox hosts with debz (you get ZFS on root, boot environments, and WireGuard that Proxmox VE doesn't provide), or run debz VMs on top of Proxmox. The kvm template turns a debz node into a libvirt KVM hypervisor — combine it with Proxmox for mixed bare-metal and VM workloads on the same WireGuard mesh.

The golden image pipeline can push qcow2 images directly to Proxmox over SSH. The Web UI has a Proxmox panel for VM lifecycle management. The two integrate cleanly without replacing each other.

I want a homelab Kubernetes cluster on old hardware. Is debz practical?

Yes, and this is one of the most natural use cases. A typical setup: one master node (any x86_64 with 4 GB RAM and 40 GB disk), two or three worker nodes (2 GB RAM minimum, 20 GB disk). Total hardware cost can be near zero if you have spare machines lying around.

Boot the USB, pick the template, wait 90 seconds. Nodes join the Salt mesh and are ready for kubeadm join automatically. No separate router, no separate DNS server, no managed switch required — WireGuard handles the overlay on whatever network you have.

Can I use debz for a CI/CD build farm?

Yes. The golden image pipeline is particularly useful here. Build one golden image per build environment definition. CoW-clone it per job — each clone boots with a unique machine-id, hostname, and WireGuard identity. Run the job. Destroy the clone. The next job gets a perfectly clean environment cloned from the same golden base in under a second.

Gitea, Woodpecker CI, and self-hosted GitHub Actions runners all work on debz worker nodes without modification. The WireGuard mesh means runners have consistent private IPs across hardware migrations.

I need an air-gapped deployment for a secured facility. Can debz handle that?

Yes — air-gap is a first-class scenario, not an afterthought. The ISO includes a complete darksite APT mirror with all packages needed for every template. Installs never phone home. The ISO itself is the entire deployment payload.

For multi-node air-gap clusters: boot the master node from ISO (which runs the full control plane in the live session), install nodes by pointing their installers at the master's darksite mirror. The WireGuard enrollment plane handles cluster formation with no internet dependency.

See Air-Gap Deployment for the full procedure.

Can I run debz on a laptop as a daily driver desktop?

Yes. The desktop profile installs GNOME, GDM, Firefox, and the Web UI. It is a standard Debian 13 GNOME desktop with ZFS on root. Boot environments mean you can upgrade packages fearlessly — a bad update is a 30-second rollback at next boot. The same snapshot policy that protects servers protects your desktop.

Hardware support is the same as any Debian 13 install. Intel and AMD graphics work out of the box. NVIDIA requires the non-free firmware package, which is in the darksite mirror.

What about edge deployments — remote sites, kiosks, embedded systems?

The air-gap installer and WireGuard mesh make debz well-suited to remote deployments. A node at a remote site installs from the USB, gets its WireGuard keys at first boot, and phones home to the master over wg1. Salt then manages it centrally the same as any other node — no VPN client to configure, no firewall rules to punch through beyond the WireGuard UDP port.

ARM64 support is in progress (v1 is AMD64 only). x86_64 industrial hardware, mini PCs, and NUCs all work well.

Can debz span multiple cloud providers simultaneously?

Yes. A debz node is a Debian 13 VM with a WireGuard interface. It runs on AWS EC2, GCP Compute, Azure VMs, Hetzner, and bare metal on the same mesh without modification. The Salt master sees all nodes regardless of where they run. Kubernetes pods can schedule across on-prem and cloud nodes on the same CNI overlay.

The Web UI cloud deploy panel handles VM provisioning on AWS, GCP, and Azure via their respective APIs. Nodes are stamped from the same golden image used for bare metal.

Installation

How long does a full install actually take?

On modern hardware with a fast NVMe: 60–90 seconds from confirming the install to power-off. On a 5400 RPM spinning disk: 3–5 minutes. The bottleneck is debootstrap unpacking packages — the ZFS pool creation and WireGuard setup are nearly instant.

First boot (firstboot service running Salt and WireGuard setup) adds another 30–60 seconds depending on the template. A full master + three workers from power-on to Kubernetes-ready is typically under 15 minutes total on fast local hardware.

Can I install non-interactively / unattended?

Yes. Pass an answers file to the installer: debz-install-target --config /path/to/answers.env. The answers file is a shell env file with variables like DEBZ_DISK, DEBZ_HOSTNAME, DEBZ_TEMPLATE, DEBZ_ZFS_ENCRYPT, etc.

The TUI installer generates this file automatically — you can run through the TUI once to generate it, then reuse it for identical deployments. The golden image pipeline bypasses the installer entirely for repeat deployments.

Can I keep my existing data / dual-boot?

The installer wipes the target disk entirely. This is intentional — a clean ZFS pool created by the installer has a known-good state. Dual-boot on the same disk is not supported.

If you need to preserve data, back it up before installing. ZFS replication (zfs send | zfs recv) is an efficient way to ship datasets to another machine before wiping.

What firmware / drivers does debz include?

The live ISO and installed system include non-free-firmware from Debian's repos — Intel WiFi, Realtek, Broadcom, AMD microcode, Intel microcode. NVIDIA firmware is in the darksite mirror but not installed by default (add it post-install or include it in your golden image).

If your hardware needs firmware that isn't loading on the live ISO, drop the .deb into the darksite pool and rebuild the ISO, or install it post-boot from the Debian repos.

Does debz support Secure Boot?

Secure Boot is supported via MOK (Machine Owner Key) enrollment. The installer can generate and enroll a signing key for the ZFS DKMS module. See Secure Boot + MOK for the procedure. If your environment mandates Secure Boot, this is the path.

For most homelab and bare-metal deployments, Secure Boot is disabled in the BIOS. debz works fine either way.

Air-Gap & Offline

What exactly is the "darksite" and what does it contain?

The darksite is a complete local APT repository baked into the ISO at /root/darksite/apt/. It contains every .deb package needed to install any debz template — base OS, ZFS, WireGuard, Salt 3006, nginx, dnsmasq, Kubernetes components, and all transitive dependencies. A dependency closure of ~2,700 packages is resolved at ISO build time and downloaded into the pool.

The installed system is configured to use the darksite mirror first. If a package isn't there, apt falls through to the Debian mirrors. In a true air-gap environment, the Debian mirrors are simply unreachable and only packages in the darksite are available — which is all you need for a complete debz install.

How do I update packages on an air-gapped cluster?

Rebuild the ISO with an updated darksite (re-run ./deploy.sh build on a machine with internet access, which re-downloads the latest versions of all packages). Deploy the new ISO to a USB stick. Boot any node from it — the live session's darksite serves the updated packages. Run apt upgrade on each node using the live ISO as the APT source.

For more controlled updates, use the golden image pipeline: build a new golden image from the updated ISO, validate it, then roll it out to nodes via CoW clone + boot environment switch.

vs. Everything Else

How is debz different from Talos Linux?

Talos is immutable and Kubernetes-only. It has no shell, no SSH, no package manager. That is powerful in a narrow context — a dedicated Kubernetes cluster where you never need to touch the node directly. The moment you want to run something that isn't a Kubernetes workload (a KVM hypervisor, a ZFS NAS, a build machine), Talos is wrong for the job.

debz is mutable, general-purpose, and SSH-accessible. You can run Kubernetes on it, or not. You can install packages, run arbitrary daemons, use it as a desktop. The tradeoff is that you bear more responsibility for keeping the OS well-configured — which is exactly what ZFS snapshots, boot environments, and Salt are for.

How is debz different from Proxmox VE?

Proxmox is a hypervisor appliance. Its OS layer is not something you are meant to modify, run arbitrary workloads on, or replicate to bare metal as a general-purpose server. It does one thing (KVM + LXC management) exceptionally well.

debz is a general-purpose host OS that can run a KVM hypervisor (the kvm template) as one of many roles. You can run debz on top of Proxmox (as a VM), alongside Proxmox (on other bare-metal nodes), or instead of Proxmox if you don't need its web UI and want a more composable setup.

How is debz different from NixOS?

NixOS is declarative and reproducible at the cost of learning an entirely new configuration language and mental model. The reproducibility guarantees are stronger in theory — but the learning curve is steep and the ecosystem has real-world rough edges with non-Nix software.

debz is reproducible through a different mechanism: a fixed, well-tested ISO, deterministic ZFS layout, and idempotent firstboot scripts. It does not require you to learn Nix. Every standard Debian/Linux skill transfers directly. The tradeoff is that debz is less hermetically sealed — but for most real infrastructure, that is fine.

How is debz different from Ubuntu Server?

Ubuntu Server is a solid general-purpose distro, but it ships with ext4 by default, no boot environments, no WireGuard mesh, no snapshot policy, and no cluster tooling. Getting from a fresh Ubuntu install to what debz gives you out of the box takes hours of manual configuration that you have to maintain across every machine.

debz is also Debian, not Ubuntu — no snap, no Canonical data collection, no LTS-vs-non-LTS confusion. Debian trixie (13) is the base, with the upstream Debian package ecosystem.

Miscellaneous

Is debz open source?

Yes. The build system, installer, snapshot tools, boot environment manager, firstboot scripts, and Web UI are all open source. The ISO is built entirely from Debian packages and open-source components.

What architectures are supported?

AMD64 (x86_64) in v1. ARM64 (aarch64) support is in progress — the build system is architecture-aware and the installer has no architecture-specific code, but the ISO build pipeline and darksite package sets need ARM64 variants of the ZFSBootMenu EFI binary and a few other components. ARM64 will land in a future release.

Can I contribute or customize the ISO for my organisation?

Yes. The build system is designed to be forked and customized. Add package sets to build/darksite/config/package-sets/, add hooks to live-build/config/hooks/, add your own firstboot template scripts to live-build/config/includes.chroot/usr/local/sbin/debz-firstboot-<template>, and rebuild. The deploy.sh script handles the full pipeline.

For organisation-specific customization, the most common approach is to add a private target-org.txt package set with your internal tools, a custom firstboot template that registers nodes with your existing Salt/Ansible master, and a golden image built from your customized ISO.

The name — why "debz"?

Debian + z (for ZFS). The z at the end also nods to ZFSBootMenu, WireGuard (which uses a lot of elliptic curve math that sounds like the end of the alphabet), and the idea that this is the last layer of infrastructure you should have to think about — after this, z, done.

Where do I go if I have a question not answered here?

Open an issue or start a discussion on the project repository. The debz.ca site links to the repo. For urgent production issues, the Installation Guide and Boot Environments docs cover the most common recovery scenarios.
