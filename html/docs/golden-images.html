Golden Images

A golden image is a fully installed, role-configured qcow2 disk image built headlessly in QEMU. Stamping clones each golden image using copy-on-write — each clone is ~1 MB at creation and grows only as data is written. No per-node install needed.

Why golden images?

Installing debz from ISO to each node individually takes 10–20 minutes per machine. A golden image is built once per template, then stamped into as many identical clones as needed in seconds. Each clone gets its own hostname, node index, machine-id, and WireGuard identity injected at stamp time — without booting the image.

ApproachTime per nodeIdentical baseUnique identity
ISO install per node10–20 minDrift riskYes
Golden + stamp<30 sec cloneGuaranteed identicalYes (injected)

Step 1 — Build a golden image

Build a golden image for a specific template on the MASTER node (or any machine with QEMU installed):

# Build a KVM golden image (uses debz-trixie-amd64-server.iso by default)
./deploy.sh golden TEMPLATE=kvm

# Build a storage node golden image
./deploy.sh golden TEMPLATE=storage

# Custom size and explicit ISO path
./deploy.sh golden TEMPLATE=master \
  GOLDEN_SIZE=60G \
  GOLDEN_ISO=./live-build/output/debz-trixie-amd64-server.iso

What the build command does

  1. 1
    Creates a GOLDEN_SIZE qcow2 disk at /var/lib/debz/golden/
  2. 2
    Assembles a FAT32 DEBZ-SEED disk image containing an answers.env with the template profile, a fixed hostname (golden-<template>), and DEBZ_ZFS_TOPOLOGY=single
  3. 3
    Boots QEMU headlessly with the debz ISO, the target disk, and the DEBZ-SEED disk attached. The live session detects the DEBZ-SEED and runs the unattended installer automatically
  4. 4
    After install completes, QEMU reboots into the freshly installed system. debz-apply-role --role <template> runs at first boot to configure template-specific services and packages
  5. 5
    The system powers off cleanly. The qcow2 at /var/lib/debz/golden/golden-<template>.qcow2 is the golden image
QEMU must be installed on the build machine. The build runs as the current user. If building on the MASTER node, QEMU is already present. If building on a laptop or CI runner, ensure qemu-system-x86 (or qemu-system-aarch64 for ARM64) is installed.

Step 2 — Stamp clones

Stamping creates CoW clones of a golden image and injects unique identity into each one without booting. Each clone is created with qemu-img create -b (backing file) and gets hostname, node index, machine-id, and hub.env injected via qemu-nbd + ZFS import into a temporary namespace. 

# Stamp 4 KVM clones starting at index 2, hostname kvm-N
./deploy.sh stamp \
  TEMPLATE=kvm \
  STAMP_COUNT=4 \
  STAMP_BASE_HOSTNAME=kvm \
  STAMP_BASE_INDEX=2

# Results in:
#   /var/lib/debz/clones/kvm-02.qcow2  (index 2, hostname kvm-02)
#   /var/lib/debz/clones/kvm-03.qcow2
#   /var/lib/debz/clones/kvm-04.qcow2
#   /var/lib/debz/clones/kvm-05.qcow2

# Custom output directory
./deploy.sh stamp TEMPLATE=storage STAMP_COUNT=2 \
  STAMP_OUTPUT_DIR=/mnt/nas/clones/

What debz-stamp-identity injects

Step 3 — Spawn VMs

Spawn clones as VMs on local libvirt or Proxmox:

# Spawn all stamped clones on the local libvirt host
./deploy.sh spawn TEMPLATE=kvm

# Spawn on a specific Proxmox host
./deploy.sh spawn TEMPLATE=storage PROXMOX_HOST=proxmox-01.debz.local

# Spawn with a custom VM ID base (Proxmox VMID starts at 200)
./deploy.sh spawn TEMPLATE=kvm PROXMOX_HOST=proxmox-01 VMID=200

Build all four templates

# Build golden images for all four templates sequentially
for template in master kvm storage vdi; do
  ./deploy.sh golden TEMPLATE=$template
done

# Verify all four exist
ls -lh /var/lib/debz/golden/

Troubleshooting

QEMU not installed

# Debian/Ubuntu
apt install qemu-system-x86 qemu-utils

# Check QEMU accelerator
qemu-system-x86_64 -accel help

DEBZ-SEED not detected during build

The live session scans for a FAT32-labeled disk named DEBZ-SEED at boot. If the build hangs at the live login prompt, the seed disk was not found:

# Check that the seed disk was created
ls -lh /var/lib/debz/seed/

# Verify FAT32 label (should output DEBZ-SEED)
blkid /var/lib/debz/seed/debz-seed-kvm.img

# Inspect seed disk contents
mcopy -i /var/lib/debz/seed/debz-seed-kvm.img -s ::/ /tmp/seed-check/

Identity injection errors

Injection uses qemu-nbd to mount the clone and import ZFS pools into a temporary namespace. Errors here usually mean qemu-nbd is not loaded:

# Load the nbd kernel module
modprobe nbd max_part=8

# Check /dev/nbd* devices exist
ls /dev/nbd*
