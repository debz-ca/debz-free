ZFS Encryption

debz uses ZFS native encryption — AES-256-GCM with a user-supplied passphrase. The passphrase is entered at the ZFSBootMenu screen before the OS loads.

How ZFS native encryption works

ZFS native encryption (introduced in OpenZFS 0.8) encrypts data at the dataset level. Each dataset has an encryption key derived from a user-supplied passphrase via PBKDF2. The data key is stored in encrypted form in the pool — it can only be decrypted with the correct passphrase.

Unlike LUKS (which encrypts the entire block device), ZFS encryption:

  • Allows per-dataset encryption settings — you can encrypt some datasets but not others
  • Encrypts data and file metadata, but not dataset names or pool structure
  • Works transparently with ZFS send/recv for encrypted replication
  • Does not encrypt the pool’s header — pool existence is visible, contents are not

Enabling encryption at install time

During guided install, choose ZFS native encryption at Step 3 and enter your passphrase. For unattended installs:

DEBZ_ZFS_ENCRYPT=1
DEBZ_ZFS_PASSPHRASE=your-strong-passphrase

The installer sets these ZFS properties on rpool:

encryption=aes-256-gcm
keyformat=passphrase
keylocation=prompt

keylocation=prompt means ZFS will request the passphrase interactively when the pool is imported — which happens at ZFSBootMenu before the OS boots.

Warning There is no passphrase recovery mechanism. If you lose the passphrase, the data is unrecoverable. Store your passphrase in a password manager. For operational environments, consider a key escrow process or use a hardware security module.

Passphrase entry at boot

At boot, ZFSBootMenu appears before the OS loads. If the pool is encrypted, it prompts for the passphrase before listing boot environments: 

ZFSBootMenu — debz
───────────────────────────────────
rpool is encrypted. Enter passphrase to unlock:
Passphrase: ████████████████

Unlocking rpool... done.

Boot environment:
  ►  rpool/ROOT/prod-node-01    [2026-03-15]
     rpool/ROOT/prod-node-01@pre-upgrade-2026-03-12

After unlocking, ZFSBootMenu can list and boot into any boot environment on the pool. The passphrase unlocks the pool key; the key is kept in memory and used for all I/O until the next reboot.

Encryption scope

In a debz encrypted install, rpool is created with encryption enabled. All datasets inherit the encryption property from the pool root.

Pool / DatasetEncrypted?Reason
rpoolYes (root key)All data encrypted
rpool/ROOT/<hostname>Yes (inherited)OS filesystem
rpool/homeYes (inherited)User home directories
rpool/srvYes (inherited)Service data
rpool/var/*Yes (inherited)Logs, cache, tmp

Checking encryption status

# Check encryption properties on all datasets
zfs get encryption,keyformat,keystatus rpool

NAME               PROPERTY    VALUE          SOURCE
rpool              encryption  aes-256-gcm    local
rpool              keyformat   passphrase     local
rpool              keystatus   available      -
rpool/ROOT         encryption  aes-256-gcm    inherited from rpool
rpool/ROOT/prod-node-01  keystatus  available  -

# Check the encryption algorithm and key length
zfs get encryption rpool
rpool  encryption  aes-256-gcm  local

Changing the passphrase

# Change the passphrase on a running system
zfs change-key rpool

Enter new passphrase: ████████████████
Re-enter new passphrase: ████████████████
Key changed for rpool.

The change takes effect immediately. No reboot is required. The next boot will prompt for the new passphrase.

Unlocking a pool manually (recovery scenario)

If you boot into the live ISO for recovery and need to access an encrypted pool: 

# Import the pool without mounting (don't auto-unlock)
zpool import -N rpool

# Load the encryption key (will prompt for passphrase)
zfs load-key rpool

Enter passphrase for rpool: ████████████████
1 / 1 keys successfully loaded

# Now mount the datasets
zfs mount -a

# Or mount a specific dataset
zfs mount rpool/ROOT/prod-node-01

Encrypted replication

ZFS allows sending encrypted datasets in raw (still-encrypted) form. The receiving host does not need the passphrase — it stores the ciphertext and can only decrypt if it has the key. This is the most secure replication model. 

# Send encrypted stream — receiving host stores ciphertext only
zfs send -Rw rpool/srv@srv-20260315-144501 | \
  ssh backup-host "zfs recv backup/srv"

# The -w flag sends raw (encrypted) data
# backup-host cannot decrypt without the passphrase

# To receive and decrypt on the backup host, load the key there too
ssh backup-host "zfs load-key backup/srv"

See Replication for a complete replication setup guide.

Performance considerations

AES-256-GCM is hardware-accelerated on all modern CPUs (AES-NI on x86-64, ARMv8 Crypto Extension on ARM64). The overhead is typically 5–15% compared to unencrypted ZFS for sequential I/O, and negligible for I/O patterns that are memory-bound. 

# Verify AES-NI is available
grep aes /proc/cpuinfo | head -1
flags : ... aes sse4_2 ...

# Check ZFS encryption acceleration is being used
cat /sys/module/icp/parameters/icp_aes_impl
aesni

Encryption and snapshots

Snapshots of encrypted datasets are themselves encrypted — the snapshot holds the same key as the live dataset. You can roll back, clone, and send snapshots normally. The passphrase does not need to be re-entered for snapshot operations on an already-unlocked pool.

Tip APT snapshots and the /srv timer operate identically on encrypted datasets — no changes to the snapshot policy are needed. The encryption is transparent to the snapshot mechanism.
