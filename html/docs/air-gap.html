Air-Gap Deployment

The debz ISO contains a complete embedded APT mirror — the darksite. Boot, install, and run fully offline. No internet connection required at any point.

What is the darksite?

The darksite is an APT mirror embedded directly in the live ISO. It runs as a local HTTP service inside the live session at http://127.0.0.1:3142/apt. The installer queries this endpoint during bootstrap and package installation, making the entire install process work with zero outbound network connectivity.

The darksite contains the full package set needed to bootstrap Debian trixie for both the server and desktop profiles, including ZFS, the kernel, eBPF tools, WireGuard, and all profile dependencies. It is a snapshot of the Debian archive at ISO build time.

Mirror auto-detection

The installer performs a health check at startup to determine which mirror to use:

  1. Check local darksite — curl http://127.0.0.1:3142/apt/ If the darksite responds with HTTP 200, the installer uses it as the primary mirror.
  2. Fall back to internet — If the darksite is not available (e.g., started from a custom live session), the installer falls back to mirror.it.ubc.ca/debian as the internet mirror.

This means the same answer file and the same installer invocation works in both online and offline environments — no conditional logic in your automation. 

# The installer reports which mirror it will use
debz-installer --guided

[debz-installer] Checking for local darksite mirror...
[debz-installer] Darksite online: http://127.0.0.1:3142/apt
[debz-installer] Using local mirror — no internet required
# Or when darksite is unavailable
[debz-installer] Checking for local darksite mirror...
[debz-installer] Darksite not found — using internet mirror
[debz-installer] Mirror: http://mirror.it.ubc.ca/debian

Fully offline install walkthrough

  1. Write the ISO to USB — The ISO is self-contained. No additional files needed.
  2. Transport the USB — physically carry it to the air-gapped location. The ISO is signed; verify the checksum before you go.
  3. Boot the live session — The darksite service starts automatically. No network configuration is required for the install itself.
  4. Run the installer — guided or unattended, it will use the local mirror.
  5. Set static IP during install — if the installed system will be accessed via local network, configure a static IP in the installer’s network step.
Note The network step in the installer configures the installed system, not the live session. You can install debz on a machine with no network cable plugged in at all — choose network method dhcp and configure it post-install, or skip network config and set it up via the console after first boot.

Unattended air-gap deployment

For deploying multiple machines in an isolated environment, use an answer file. The only per-machine variable that changes is the hostname and IP: 

# node-answers.env — template for isolated network deployment
DEBZ_HOSTNAME=node-01         # change per machine
DEBZ_PROFILE=server
DEBZ_USERNAME=admin
DEBZ_PASSWORD=vault-managed

DEBZ_DISK=/dev/sda
DEBZ_ZFS_TOPOLOGY=single
DEBZ_ZFS_ENCRYPT=1
DEBZ_ZFS_PASSPHRASE=site-wide-passphrase

DEBZ_TIMEZONE=UTC
DEBZ_LOCALE=en_US.UTF-8
DEBZ_KEYBOARD_LAYOUT=us

DEBZ_NET_METHOD=static
DEBZ_NET_IFACE=eth0
DEBZ_NET_IP=10.10.0.11      # change per machine
DEBZ_NET_PREFIX=24
DEBZ_NET_GW=10.10.0.1
DEBZ_NET_DNS=10.10.0.1      # local DNS only

What packages are in the darksite?

The darksite includes packages for both profiles plus all installer dependencies:

Core (both profiles)

  • Debian trixie base system (debootstrap output)
  • linux-image, linux-headers
  • zfs-dkms, zfs-initramfs, zfsutils-linux
  • zfsbootmenu, grub-efi-amd64
  • openssh-server, sudo
  • wireguard-tools, wireguard
  • bpftool, bpftrace, bcc-tools
  • nftables, chrony
  • systemd-resolved, network-manager
  • shim-signed, mokutil

Desktop (additional)

  • GNOME Shell, GDM3
  • Firefox ESR
  • GNOME Terminal, Files, Settings

Package freshness

The darksite is a snapshot of the Debian archive at the time the ISO was built. Packages in the darksite are the versions available at build time — they may not be the latest available from Debian.

For security-sensitive air-gap deployments, update packages after installation using a managed update process — either bring an updated offline mirror on USB, use a local Debian mirror inside the network perimeter, or synchronise a Squid proxy cache for controlled internet access to security.debian.org.

Warning The built-in darksite is a snapshot baked at ISO build time. To add new packages at runtime, use a DEBZ-DARK disk (see below) rather than modifying the live squashfs.

DEBZ-DARK: bring your own repo disk

For sites with large or frequently-updated package sets, container images, Helm charts, or CI/CD payloads that are too large to bake into the ISO, debz supports a companion disk labeled DEBZ-DARK. At boot, the live system automatically detects and mounts it, merging its contents into /root/darksite/ before the APT mirror or installer starts.

A DEBZ-DARK disk can be any size — tested up to 8.8 TB. Format it as exFAT (recommended — readable on macOS, Windows, and Linux), EXT4, or NTFS, then set the filesystem label to DEBZ-DARK.

Disk layout

DEBZ-DARK:/
├── apt/          # .deb files → merged into darksite APT repo
├── images/       # OCI/docker images → available at /root/darksite/images/
├── helm/         # Helm charts → available at /root/darksite/helm/
├── ansible/      # Ansible playbooks and roles
├── salt/         # Salt states and pillars
├── iso/          # OS images for golden image builds
├── pipelines/    # CI/CD scripts and pipeline definitions
└── custom/       # Anything else — bind-mounted as-is

Any directory not in the list above is also automatically bind-mounted under /root/darksite/<name>.

Preparing a DEBZ-DARK disk

# Format as exFAT and label it DEBZ-DARK (Linux)
mkfs.exfat -n DEBZ-DARK /dev/sdX

# Or on macOS
diskutil eraseDisk ExFAT DEBZ-DARK /dev/diskN

# Copy your packages, images, and payloads
cp *.deb  /media/DEBZ-DARK/apt/
cp *.tar  /media/DEBZ-DARK/images/
cp *.tgz  /media/DEBZ-DARK/helm/

# Plug into any debz machine and reboot

What happens at boot

  1. debz-darksite-extend.service starts before the APT mirror.
  2. It scans for a disk labeled DEBZ-DARK using blkid.
  3. Mounts it read-only at /mnt/debz-dark.
  4. Bind-mounts known subdirectories into /root/darksite/.
  5. For apt/: copies .deb files into the darksite pool and rebuilds the package index with dpkg-scanpackages.
  6. Logs everything to /var/log/debz/darksite-extend.log.
# Check DEBZ-DARK mount status
systemctl status debz-darksite-extend.service

# See what was merged
journalctl -u debz-darksite-extend --no-pager

# Inspect the extended darksite
ls /root/darksite/
APT repo rebuild — When the DEBZ-DARK apt/ directory contains .deb files, debz automatically runs dpkg-scanpackages to regenerate the Packages index. Your custom packages are immediately installable via apt install <package> without any additional configuration.

Post-install APT configuration (air-gap)

After installation, if the system remains air-gapped, configure APT to point to your local mirror instead of the internet: 

# Replace internet mirrors with local mirror
cat > /etc/apt/sources.list <<EOF
deb http://apt.local.internal/debian trixie main contrib non-free non-free-firmware
deb http://apt.local.internal/debian-security trixie-security main
deb http://apt.local.internal/debian trixie-updates main
EOF

apt update

Darksite service details

In the live session, the darksite is managed by a systemd service. You can check its status or restart it if needed: 

# Check darksite service (on live ISO)
systemctl status debz-darksite.service
