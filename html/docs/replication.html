Replication

Any debz node can stream datasets to any ZFS-capable host using zfs send and zfs recv. Combine with WireGuard for encrypted streams over untrusted networks.

How ZFS replication works

zfs send produces a binary stream of a dataset’s contents (or the delta between two snapshots). zfs recv on the receiving host applies that stream to recreate the dataset exactly. Because the stream is deterministic and snapshot-based, incremental replication sends only changed blocks — typically a fraction of the dataset size.

Supported destinations from debz:

  • Another debz node
  • TrueNAS Scale or Core (both use OpenZFS)
  • Any Linux host with OpenZFS installed
  • A cloud VM (AWS, Hetzner, etc.) with OpenZFS

Prerequisites

On the sending debz node:

  • SSH key-based access to the receiving host (no password)
  • A snapshot on the dataset to replicate

On the receiving host:

  • OpenZFS installed
  • A pool to receive datasets into
  • SSH access from the sending node

Initial replication (full send)

# Take a snapshot to send
zfs snapshot -r rpool/srv@repl-20260315

# Full send to a remote host
zfs send -R rpool/srv@repl-20260315 | \
  ssh backup-host "zfs recv -F backup/prod-node-01/srv"

# -R sends recursively (all child datasets)
# -F on recv forces rollback if needed

Incremental replication

Once the initial send is complete, subsequent sends only transfer blocks changed since the last common snapshot. This is the efficient, ongoing replication pattern: 

# Take a new snapshot
zfs snapshot -r rpool/srv@repl-20260316

# Send only the delta between the two snapshots
zfs send -R -I rpool/srv@repl-20260315 rpool/srv@repl-20260316 | \
  ssh backup-host "zfs recv -F backup/prod-node-01/srv"

# -I repl-20260315 = send delta from that snapshot to @repl-20260316
Tip Use mbuffer between the send and receive pipes to smooth out I/O spikes and provide a progress indicator: zfs send ... | mbuffer -q -s 128k -m 1G | ssh ... "mbuffer -q -s 128k | zfs recv ..."

Encrypted replication (raw send)

For encrypted pools, use -w (raw send) to transmit the ciphertext without decrypting on the sending host. The receiving host stores encrypted data and cannot decrypt without the pool passphrase. 

# Raw encrypted send — ciphertext only, no decryption on either end
zfs send -Rw rpool/srv@repl-20260315 | \
  ssh backup-host "zfs recv backup/prod-node-01/srv"

# Incremental raw send
zfs send -Rw -I rpool/srv@repl-20260315 rpool/srv@repl-20260316 | \
  ssh backup-host "zfs recv backup/prod-node-01/srv"

If the receiving host needs to read the data (not just store it), load the key on the receiving side after receiving the dataset: 

# On receiving host — load key after raw recv
ssh backup-host "zfs load-key backup/prod-node-01/srv"

Replication over WireGuard

WireGuard is built into the debz kernel. For replication over untrusted networks (internet, shared data centre fabric), set up a WireGuard tunnel between the nodes and send ZFS streams over it. SSH over WireGuard provides both encryption and authentication.

Quick WireGuard tunnel setup

# On sending node (prod-node-01) — generate keys
wg genkey | tee /etc/wireguard/wg0.key | wg pubkey > /etc/wireguard/wg0.pub

# /etc/wireguard/wg0.conf on prod-node-01
[Interface]
PrivateKey = <prod-node-01 private key>
Address = 10.200.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <backup-host public key>
AllowedIPs = 10.200.0.2/32
Endpoint = backup-host.example.com:51820

# Bring up the tunnel
wg-quick up wg0

# Verify
wg show
# Replicate over WireGuard (SSH via wg0 IP)
zfs send -Rw -I rpool/srv@repl-20260315 rpool/srv@repl-20260316 | \
  ssh 10.200.0.2 "zfs recv backup/prod-node-01/srv"

Automated replication with systemd

Set up a systemd timer to run incremental replication daily:

# /usr/local/sbin/debz-replicate-srv
#!/bin/bash
set -euo pipefail

DATASET=rpool/srv
REMOTE=backup-host
REMOTE_DATASET=backup/prod-node-01/srv
SNAP_PREFIX=repl

latest_local=$(zfs list -t snapshot -H -o name -s creation "$DATASET" \
  | grep "@$SNAP_PREFIX-" | tail -1 | cut -d@ -f2)

new_snap="$SNAP_PREFIX-$(date -u +%Y%m%d-%H%M%S)"
zfs snapshot -r "$DATASET@$new_snap"

if [ -n "$latest_local" ]; then
  zfs send -Rw -I "$DATASET@$latest_local" "$DATASET@$new_snap" \
    | ssh "$REMOTE" "zfs recv -F $REMOTE_DATASET"
else
  zfs send -Rw "$DATASET@$new_snap" \
    | ssh "$REMOTE" "zfs recv -F $REMOTE_DATASET"
fi
# /etc/systemd/system/debz-replicate-srv.service
[Unit]
Description=debz /srv replication
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/local/sbin/debz-replicate-srv

# /etc/systemd/system/debz-replicate-srv.timer
[Unit]
Description=debz /srv replication timer

[Timer]
OnCalendar=*-*-* 03:00:00
RandomizedDelaySec=300

[Install]
WantedBy=timers.target

# Enable and start
systemctl enable --now debz-replicate-srv.timer

Replication to TrueNAS Scale

TrueNAS Scale uses OpenZFS and supports receiving zfs send streams. Use TrueNAS’s built-in replication task UI or replicate manually: 

# Create a dataset on TrueNAS to receive into
# (Do this via TrueNAS UI: Storage → Add Dataset)

# From debz node — send to TrueNAS
zfs send -Rw rpool/srv@repl-20260315 | \
  ssh admin@truenas.local "zfs recv -F tank/backups/prod-node-01-srv"
Note TrueNAS may use a different OpenZFS version than debz. If the sending pool has newer feature flags than the receiving pool supports, the recv will fail. Check OpenZFS version compatibility before replicating across major versions. Use zfs send -p to include property metadata and inspect the stream with zstream dump to check feature requirements.

Monitoring replication

# List snapshots on both ends to verify sync
zfs list -t snapshot rpool/srv | grep repl
ssh backup-host "zfs list -t snapshot backup/prod-node-01/srv | grep repl"

# Check how much data would be sent in the next incremental
zfs send -nv -I rpool/srv@repl-20260315 rpool/srv@repl-20260316
would send: 284M
