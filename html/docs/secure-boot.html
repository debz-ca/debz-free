Secure Boot + MOK

debz generates a per-machine Machine Owner Key (MOK), signs all ZFS kernel modules, and configures DKMS to auto-sign future module builds — all during installation.

Why MOK is needed for ZFS

ZFS on Linux is out-of-tree — it is not distributed as part of the official kernel. Out-of-tree kernel modules must be signed with a key that is enrolled in the MOK database (or the platform key database) before Secure Boot will permit them to load.

Microsoft’s CA (which signs the shim bootloader) does not sign ZFS. The standard solution is to generate a Machine Owner Key, sign the ZFS modules with it, and enroll the public half in the UEFI MOK database. Shim trusts the MOK database, so a MOK-signed module loads under Secure Boot.

Important Install debz with Secure Boot disabled in firmware. The live ISO’s ZFS modules are not signed for your machine’s MOK — enabling Secure Boot before the key is enrolled will prevent ZFS from loading and the system will not boot.

What the installer does

  1. Generates RSA-2048 key pair at /etc/debz/mok/debz-mok.key (private) and /etc/debz/mok/debz-mok.crt (certificate). Key material has mode 600 and is root-owned.
  2. Signs ZFS kernel modules — all .ko, .ko.gz, and .ko.zst files in /lib/modules/ belonging to the zfs, spl, znvpair, zunicode, and zavl modules are signed with sign-file.
  3. Configures DKMS — adds a DKMS post-build hook so every future ZFS module rebuild (triggered by a kernel upgrade) is automatically signed with the same MOK key.
  4. Queues MOK enrollment — runs mokutil --import /etc/debz/mok/debz-mok.crt with a one-time password. The password is saved to /var/log/installer/mok-password.txt (mode 600).
  5. Powers off — the system shuts down. You enable Secure Boot in firmware before the next boot.

First boot — MOK enrollment

When the system boots for the first time after enabling Secure Boot, shim detects a pending MOK enrollment and launches MokManager before handing off to ZFSBootMenu.

  1. MokManager screen appears: Enroll MOK
  2. Select Continue
  3. Select Yes to view and enroll the key
  4. Enter the enrollment password — retrieve it from /var/log/installer/mok-password.txt
  5. Select Reboot

After reboot, ZFS modules load under Secure Boot. Verify:

# Confirm Secure Boot is active
mokutil --sb-state
SecureBoot enabled

# Confirm the MOK is enrolled
mokutil --list-enrolled | grep -A4 debz

# Confirm ZFS module loaded
lsmod | grep zfs
zfs                  4890624  4
zunicode              335872  1 zfs
zavl                   16384  1 zfs
znvpair               90112  2 zfs,zavl
spl                   102400  1 zfs

Retrieving the MOK password

The enrollment password is saved during installation. If you need to retrieve it after first boot, it remains on disk at /var/log/installer/mok-password.txt. 

# Only readable by root
cat /var/log/installer/mok-password.txt

The password is a one-time authentication token used during the MokManager enrollment screen. After enrollment it has no further security significance — the enrolled key itself is what Secure Boot trusts.

Kernel upgrades and DKMS auto-signing

When a new kernel is installed via APT, DKMS builds a new ZFS module for it. The DKMS post-build hook installed by debz signs the new module automatically. No manual signing is needed. 

# What happens during a kernel upgrade (automated)
Running module version sanity check.
Building module:
  Cleaning build area...
  Building for kernel 6.12.5-amd64
  Install completed.
  Signing module: /lib/modules/6.12.5-amd64/updates/dkms/zfs.ko
  Signature applied with /etc/debz/mok/debz-mok.key
Note DKMS builds occur at kernel package install time, before the initramfs is regenerated. The signed modules are included in the new initramfs automatically. If a DKMS build fails for any reason, the APT snapshot hook will have already created a pre-upgrade boot environment you can roll back to.

Re-enrolling the MOK key

If you need to re-enroll the MOK (e.g. firmware reset cleared the MOK database), generate a new enrollment request using the existing key — do not generate a new key, as existing module signatures would become invalid: 

# Re-request enrollment of the existing certificate
mokutil --import /etc/debz/mok/debz-mok.crt

# This prompts for a new one-time password
# Reboot, follow MokManager steps, enroll
systemctl reboot

Generating a new MOK key

If you need to rotate the key (e.g. key compromise), generate a new key, re-sign all current modules, configure DKMS, and re-enroll: 

# Generate new key pair
openssl req -new -x509 -newkey rsa:2048 -keyout /etc/debz/mok/debz-mok.key \
  -out /etc/debz/mok/debz-mok.crt -days 3650 -subj "/CN=debz MOK $(hostname)/" \
  -nodes

# Remove old enrollment, enroll new key
mokutil --delete /etc/debz/mok/debz-mok-old.crt
mokutil --import /etc/debz/mok/debz-mok.crt

# Re-sign all ZFS modules with new key
for ko in $(find /lib/modules -name "*.ko" -path "*/zfs/*"); do
  /usr/lib/linux-kbuild-*/scripts/sign-file sha256 \
    /etc/debz/mok/debz-mok.key \
    /etc/debz/mok/debz-mok.crt \
    "$ko"
done

# Rebuild initramfs to include newly signed modules
update-initramfs -u -k all

systemctl reboot
Warning After generating a new key, you must delete the old enrolled key in MokManager during the next boot and enroll the new one. Until the old key is deleted, either key works. Having two enrolled MOK keys is safe but may confuse auditing tools.

Secure Boot in Proxmox VMs

Proxmox 8.x supports OVMF (UEFI firmware) with Secure Boot. To enable Secure Boot for a debz VM:

  1. In VM hardware settings, set BIOS: OVMF (UEFI)
  2. Add an EFI disk (required for OVMF)
  3. Check Pre-Enroll keys (loads Microsoft CA — required for shim)
  4. Check Secure Boot

Install debz with Secure Boot unchecked in the VM hardware settings, complete the MokManager enrollment on first boot, then verify Secure Boot is working as described above. See Proxmox + debz for a full VM setup walkthrough.
