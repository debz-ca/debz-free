debz / docs / Web UI

Web UI

debz includes a browser-based management interface backed by a Python 3 asyncio server. It provides install wizards, real-time status panels, ZFS and Kubernetes management, and live log streaming — all over a WebSocket API.

Screenshots

Every screenshot is the real running UI on live hardware.

Installer — five steps, under 90 seconds, fully unattended option

Five screens: Template (MASTER / KVM / STORAGE / VDI), Disk (auto-detected with size and model), Identity (hostname, cluster index, WireGuard plane assignments), Options (ZFS encryption, darksite payload, mirror vs. single disk), and Confirm (full summary before any write). The same install runs headlessly via answers.env. No Anaconda. No Calamares. No compiled binary. Every step is a standard Debian shell operation — fully auditable by anyone with a shell.

5-step wizard answers.env headless debootstrap zfsbootmenu fully auditable no vendor sdk
Spawn — launch a VM or microcontainer from any browser

Spawning creates a new VM or Firecracker microVM on any KVM host reachable over the WireGuard mesh. Pick type, template, target host, name, count, vCPUs, RAM, and disk. debz stamps a ZFS CoW clone from the matching golden image, injects a unique identity (hostname, machine-id, WireGuard keys), and boots it in under 30 seconds.

Firecracker microVMs cold-start in under 125 ms. Clone 50 from a golden rootfs in a minute. Each costs ~1 MB on disk until it diverges. Use them for ephemeral CI jobs, serverless-style workloads, or high-density function runners.

One-time prerequisite per KVM/Proxmox host: SSH key-based root access from the MASTER node.
ssh-copy-id root@<your-kvm-or-proxmox-host>
kvm · qemu firecracker microvm cow clone 125ms cold-start ssh-copy-id prereq
Golden Images — build once, stamp everywhere, darksite-ready

A golden image is a ZFS boot environment snapshot of a fully-configured node. Stamp hundreds of CoW clones from it instantly — each costs ~1 MB until it diverges, takes under 5 seconds, and gets a unique hostname, machine-id, and WireGuard identity. No cloud-init. No Packer.

The darksite pipeline: populate darksite/ with private .deb packages, OCI container images, Helm charts, Ansible playbooks, and postinstall.sh before building. They are baked into every stamped clone. No internet, no registry, no artifact server required at deploy time. Every package is a standard Debian .deb. Every step is a shell script. 100% auditable.

zfs snapshot cow clone stamping darksite payload air-gap ready no packer 100% auditable
WireGuard — four planes, live peer management

Four interfaces (wg0–wg3) with deterministic IPs per cluster index. Peer table with handshake age and transfer bytes. Add or remove peers live without restarting the interface.

wg0–wg3 deterministic IPs live peer add/remove
Salt — fleet management, your states on top

Every node self-registers via cert-authenticated auto-accept on firstboot. Fleet table with last-seen, role, and highstate status. debz uses Salt for WireGuard key distribution and K8s join tokens — not your workloads. Bring your own states via the darksite payload.

cert auth auto-register highstate fleet
Kubernetes — workers auto-join in 90 seconds

KVM nodes fetch the join token from Salt mine and run kubeadm join automatically on first boot. Cilium CNI over wg2. Hubble observability. No cloud load balancer, no NodePort gymnastics.

cilium · wg2 auto-join hubble
ZFS — pool health, datasets, snapshots

Pool RAID topology, health state, error counts, full dataset tree with used/available sizes. Snapshot rollback and clone from the panel. Deterministic layout identical on every node in the estate.

pool health dataset tree snapshot rollback
Virtual Machines — zvol-backed, instant snapshots

Running and stopped VMs with start / stop / reboot / destroy actions. Backed by ZFS zvols — CoW snapshots, compression, and replication included automatically.

Every interface — physical, bridge, VLAN, all four WireGuard planes — with addresses, MTU, and link state as reported by the kernel. Not config-file assumptions.

Logs — real-time tail, every source

System journal, installer, Salt, snapshots, firstboot — streamed live over WebSocket. Switch sources instantly. Installer logs preserved after install. Audit trail (debz-audit) streams here too.

Live session — full GNOME, no install required

The ISO boots into a complete GNOME desktop in RAM. Identical to the installed system. Burn to USB or load in a hypervisor — the target becomes a live desktop and installable node immediately.

Cluster Designer — full topology in one view

Define node groups by role, count, vCPU, RAM, disk, and WireGuard planes. The topology canvas renders the full cluster map live as you design. Load a built-in template (SCC 6-node, HA Production 16-node k8s, k8s Minimal) or import any saved .json. Every design includes firewall rules and enabled services — hit Export to get a single shareable file. Hit Deploy to route straight to the Cloud provisioner.

role-aware topology wg plane assignment import / export json scc · ha-k8s templates
Firewall Designer — zone rules + live nftables preview

Zone-based rule table (external / dmz / internal / wg / mgmt) with WireGuard plane badges showing which tunnel each rule targets. Add, delete, and edit rules inline. Hit Preview nftables to generate a valid nftables.conf from the current ruleset — copy it directly to clipboard. Templates load their full firewall policy automatically (SCC: 8 rules, HA k8s: 13 rules including 6443, etcd 2379/2380, NodePorts 30000–32767).

zone-based rules wg-plane aware live nftables preview copy to clipboard
Services — stack configurator

Toggle 18 services across 5 categories (Security, Automation, Monitoring, Kubernetes, Data, Management). Templates pre-select the right stack for the role. Enabled services feed the deployment plan.

Panels

Dashboard

Dashboard

The landing panel. Stat cards show: hostname, node role/template, CPU count, RAM, node index, wg1 IP. A mini WireGuard table shows all four planes with their current state (up/down) and peer count. Quick-action buttons: reboot, shutdown, run highstate, open logs.

Install

Install

A 5-step guided wizard that mirrors the TUI installer. Progress is streamed in real time via WebSocket. The steps are:

  1. Template — select master / kvm / storage / vdi with description cards
  2. Disk — disk list auto-populated from lsblk, topology selector, encryption toggle
  3. Identity — hostname, username, password, SSH public key (optional)
  4. Options — timezone, locale, keyboard, network method, NVIDIA/eBPF toggles
  5. Confirm — summary of all choices, confirmation checkbox, Install button that starts the installer process with real-time log streaming
WireGuard

WireGuard

Four plane cards (wg0–wg3) each showing: interface state, assigned IP, listen port, and number of active peers. Below the cards, a peer table for wg1 shows all enrolled nodes with their last-handshake time, allowed IPs, and a “remove peer” action. The panel auto-refreshes every 5 seconds via WebSocket.

Network

Network

Interface table showing all network interfaces (physical and virtual): name, state, MAC, IP addresses. Includes WireGuard interfaces alongside Ethernet. DHCP lease information is shown where available via NetworkManager.

Golden Images

Golden Images

Per-template status cards showing whether a golden image exists, its size, and last build time. Build and stamp buttons trigger the corresponding deploy.sh commands with real-time output streamed to the panel. Stamp form: template, count, base hostname, base index.

Spawn

Spawn

Form for spawning VMs, microVMs (Firecracker), containers, or Kubernetes pods from golden image clones. Options: template, target host (local libvirt or Proxmox), count, resource overrides (CPU/RAM/disk). Spawn jobs are shown with live status.

VMs

VMs

libvirt VM table with columns: name, state, CPUs, RAM, disk. Actions per VM: start, shutdown, force-stop (destroy), and delete (with disk). A “new VM” button opens the spawn wizard pre-filled for local libvirt.

ZFS

ZFS

Pool health overview with usage bars per pool. Dataset browser showing the full dataset tree with mountpoint, used, available, and compression ratio. Actions: create snapshot, list snapshots for a dataset, rollback, promote clone. Boot environment panel mirroring debz-be list with activate and rollback buttons.

Kubernetes

Kubernetes

Node table with columns: name, role (control-plane/worker), status, wg2 IP, Kubernetes version, pods running. Actions: join-as-worker (runs debz-k8s-join on a remote node via Salt), drain, cordon, uncordon. A pod count sparkline per node.

Salt

Salt

Minion table: minion ID, wg1 IP, node role, node index, last highstate time, status. Actions per minion: run highstate, ping, remove key. Cluster-wide actions: highstate-all, ping-all, accept-all-pending. Pending key list with accept/reject per key.

Logs

Logs

Real-time log tailing via WebSocket. A dropdown selects which log file to tail. Available logs: installer, storage, ZFS, network, security, bootstrap, snapshots, firstboot, autobootstrap. A filter field matches lines by regex. ANSI color codes are rendered (green for OK, yellow for warnings, red for errors).

WebSocket API

All UI actions go through the WebSocket at ws://<host>:8081/ws. Messages are JSON with an action field and optional parameters. Responses stream back as JSON events with type: "data" | "progress" | "done" | "error".
