ZFS Dataset Layout

Every debz installation uses the same deterministic ZFS dataset hierarchy. Automation, replication, and snapshot scripts work identically across every node.

Complete layout

# Root pool — all OS and data datasets

rpool                             canmount=off, mountpoint=none
rpool/ROOT                        canmount=off, mountpoint=none
rpool/ROOT/<hostname>             mountpoint=/  (ZFSBootMenu)
rpool/home                        mountpoint=/home
rpool/root                        mountpoint=/root  (root user home)
rpool/srv                         mountpoint=/srv   (snapshotted every 15 min)
rpool/var                         canmount=off, mountpoint=none
rpool/var/log                     mountpoint=/var/log
rpool/var/cache                   mountpoint=/var/cache
rpool/var/tmp                     mountpoint=/var/tmp

Why rpool-only?

debz uses ZFSBootMenu as the bootloader — a standalone EFI binary that lives in the EFI system partition. ZFSBootMenu can read the full rpool with all modern ZFS features enabled, locate boot environments under rpool/ROOT/*, and hand off directly to the kernel.

Because ZFSBootMenu does not rely on GRUB2 to read ZFS, there is no need for a separate boot pool (bpool) with restricted feature flags. The EFI system partition holds the ZFSBootMenu binary, and rpool holds everything else.

Why is the hostname in the dataset path?

rpool/ROOT/<hostname> follows the OpenZFS community convention for boot environments. ZFSBootMenu scans */ROOT/* to find bootable environments, so the container dataset must be named ROOT.

The hostname in the leaf dataset name disambiguates when multiple systems share a pool (uncommon but valid) and makes snapshot and replication output readable without additional context.

Warning The hostname is baked into the dataset path at install time. Renaming the host in /etc/hostname does not rename the datasets. If you rename datasets post-install you must update /etc/fstab, /etc/zfs/zfs-list.cache/, and the ZFSBootMenu configuration, then regenerate the initramfs. It is easier to reinstall.

Dataset properties

DatasetKey propertiesRationale
rpool/ROOT/<hostname> mountpoint=/, compression=lz4, atime=off Root filesystem, snapshotted by APT hooks
rpool/home mountpoint=/home, compression=lz4 Excluded from root BE snapshots, survives rollbacks
rpool/root mountpoint=/root, compression=lz4 root user home, separate from /home for permissions
rpool/srv mountpoint=/srv, compression=lz4 Service data — 15-min snapshot timer, keep last 4
rpool/var canmount=off, mountpoint=none Container only — prevents /var from being mounted separately
rpool/var/log mountpoint=/var/log, atime=off Log persistence across root rollbacks
rpool/var/cache mountpoint=/var/cache, compression=lz4 APT cache survives rollbacks, no dedup needed
rpool/var/tmp mountpoint=/var/tmp, sync=disabled Temporary files — sync off is safe for /var/tmp

Why /home and /var/log are separate

Boot environment snapshots and rollbacks target rpool/ROOT/<hostname> only. The datasets mounted at /home, /var/log, /var/cache, and /var/tmp are not children of the ROOT dataset — they are siblings. This means:

  • Rolling back the OS does not destroy user home directories
  • Rolling back the OS does not erase logs from before the rollback
  • Rolling back the OS does not lose the APT cache (faster apt-get after rollback)

This is the key architectural advantage of dataset separation versus a monolithic root dataset.

Adding your own datasets

You can add datasets under rpool for your own workloads. Recommended practice:

# Add a dataset for a database
zfs create -o mountpoint=/var/lib/postgresql rpool/var/postgresql

# Add a dataset for container volumes
zfs create -o mountpoint=/var/lib/containers rpool/var/containers

# Add a dataset with custom compression
zfs create -o mountpoint=/data -o compression=zstd rpool/data

Datasets you add will persist across root boot environment rollbacks as long as they are not children of rpool/ROOT/<hostname>.

Tip Place service-specific datasets under rpool/srv/<service> or rpool/var/<service> to benefit from the existing snapshot policies. The 15-minute rpool/srv snapshot is recursive — child datasets are snapshotted automatically.

Pool topology

The installer creates rpool with the chosen topology:

zpool status rpool

# Scrub the pool (run monthly)
zpool scrub rpool

# Check pool health for automation/monitoring
zpool status -x    # prints "all pools are healthy" or details of errors
