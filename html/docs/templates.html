Node Templates

debz defines four node roles. Each installed machine runs exactly one template. Templates are selected at install time via the guided wizard, answer file, or DEBZ-SEED disk — and applied via Salt highstate after enrollment.

Template overview

Template Role Key Services WG Planes Used GPU Support
master Control plane hub Salt master, WireGuard hub, dnsmasq, nftables, KVM wg0–wg3 (hub on all) No
kvm Hypervisor QEMU/KVM, libvirt, OVMF, bridge networking wg1, wg2, wg3 Optional (PCIe passthrough)
storage ZFS NAS/SAN NFS, iSCSI (tgt), Samba, prometheus-node-exporter wg1, wg3 No
vdi Streaming desktop mediamtx, wf-recorder, FFmpeg, GNOME, optional NVIDIA wg1, wg2 Yes (NVIDIA nvenc)

Selecting a template

In the guided TUI installer, template selection is the first step. In an answer file, set DEBZ_PROFILE: 

DEBZ_PROFILE=master   # master | kvm | storage | vdi

On a DEBZ-SEED disk, include DEBZ_PROFILE in answers.env alongside disk, identity, and network settings. The debz-autoinstall service detects the FAT32 labeled disk on live boot and runs the installer non-interactively.

Note Template selection determines which Salt state tree runs at highstate. You can re-apply or change a node’s role after install with debz-apply-role --role <template>, but this is not the normal path — golden images are pre-baked per template.
Template Details

MASTER

The MASTER is the cluster control plane. A single MASTER node on a USB drive gives you an instant, portable control plane — plug in, boot, and the hub is live. No storage required for basic hub operation; the live session is fully functional as a MASTER.

What runs at boot

  • WireGuard: wg0–wg3 all up, hub IPs .1 on each plane
  • Salt master: bound to 10.78.0.1 (wg1), accepts minions after key approval
  • dnsmasq: DNS and DHCP for the cluster, serves debz.local domain
  • nftables: stateful firewall, NAT for nodes without public IPs, WG traffic rules
  • KVM/libvirt: available for running VMs directly on the master if needed
  • debz-autobootstrap: 30-second loop on wg0, issues hub.env to enrolling nodes
  • debz-webui: management UI at http://10.78.0.1:8080

SSH access

# From any node enrolled on wg1
ssh admin@10.78.0.1

# From outside the cluster (public IP, if configured)
ssh admin@<master-public-ip>

Applying the master role

debz-apply-role --role master

KVM

KVM nodes run QEMU/KVM hypervisors with libvirt management. They host the golden-image-based clones spawned by deploy.sh spawn and can also run arbitrary VMs.

Key packages

qemu-system-x86, libvirt-daemon-system, virtinst, ovmf, bridge-utils, virt-manager (on desktop profile)

Bridge networking

# KVM node: check bridge status
ip link show type bridge
virsh net-list --all

# Spawn a golden-image clone on this KVM node
./deploy.sh spawn TEMPLATE=kvm PROXMOX_HOST=kvm-01.debz.local

# Or spawn via libvirt directly
virt-install --import --name kvm-clone-01 \
  --disk /var/lib/debz/clones/kvm-clone-01.qcow2 \
  --memory 4096 --vcpus 4 --os-variant debian12

STORAGE

STORAGE nodes run ZFS as NAS and SAN, exporting data to the cluster over the storage plane (wg3, 10.80.0.0/16). They do not run Kubernetes workloads and have no desktop environment.

Exporting NFS over wg3

# Enable NFS exports on a ZFS dataset over the storage plane
zfs set sharenfs="rw=@10.80.0.0/16,no_root_squash" rpool/srv/data

# Verify the export
showmount -e localhost

# From a KVM or VDI node, mount it
mount -t nfs 10.80.0.3:/srv/data /mnt/data

iSCSI via tgt

# Create a ZFS zvol for iSCSI backing
zfs create -V 100G -s rpool/iscsi/vol0

# Configure tgt target (edit /etc/tgt/conf.d/debz.conf)
# Restart and verify
systemctl restart tgt
tgtadm --mode target --op show

prometheus-node-exporter

Bound to the wg3 interface. Scrape target for Prometheus running on the MASTER:

# /etc/default/prometheus-node-exporter on STORAGE node
ARGS="--web.listen-address=10.80.0.3:9100"

VDI

VDI nodes stream a live desktop session over the network using mediamtx as the media server. Clients connect via browser (HLS/WebRTC) or any SRT-capable player. NVIDIA GPUs are auto-detected and nvenc is used if available.

mediamtx ports

PortProtocolPurpose
8888HTTPHLS stream (browser-compatible)
8889HTTPWebRTC stream
8890UDP/TCPSRT stream (low-latency)

Starting a VDI session

# Start a desktop capture session
debz-vdi-session start

# With NVIDIA encoding (auto-detected if GPU present)
debz-vdi-capture --encoder nvenc --output srt://10.78.0.5:8890

# Connect from a browser on wg1
# http://10.78.0.5:8888/stream.m3u8  ← HLS
# http://10.78.0.5:8889/stream       ← WebRTC

# Or use VLC / ffplay with SRT
ffplay "srt://10.78.0.5:8890?mode=caller"
NVIDIA detection At role-apply time, debz-apply-role checks for NVIDIA hardware. If found, it installs the nvidia-driver package from trixie and configures wf-recorder to use nvenc. Set DEBZ_ENABLE_NVIDIA=1 in the answer file to force NVIDIA setup even before role application.
