Boot Environments

Boot environments are ZFS snapshots of rpool/ROOT/<hostname> that ZFSBootMenu can boot directly. Roll back a bad upgrade in under 30 seconds without reinstalling.

How boot environments work

A boot environment (BE) is a ZFS snapshot of the root dataset (rpool/ROOT/<hostname>) that has been cloned or is bootable in place. ZFSBootMenu — the EFI bootloader installed by debz — scans rpool/ROOT/* at boot time and presents any snapshots or clones it finds as selectable boot targets.

Because /home, /var/log, and other data datasets are not under rpool/ROOT, they are untouched by any boot environment operation. Rolling back the OS does not affect user data or logs.

debz-upgrade automatically creates a boot environment named with the current date before running apt dist-upgrade. If the upgrade breaks something, you can roll back from ZFSBootMenu without ever logging in to the broken system.

debz-be — boot environment manager

List boot environments

debz-be list
NAME                          CREATION              ACTIVE   MOUNTPOINT
rpool/ROOT/prod-node-01       2026-01-15 09:00 UTC  yes      /
rpool/ROOT/prod-node-01@pre-upgrade-2026-03-01   2026-03-01 14:22 UTC   snap   -
rpool/ROOT/prod-node-01@pre-upgrade-2026-03-12   2026-03-12 08:11 UTC   snap   -
rpool/ROOT/prod-node-01@manual-stable            2026-02-20 11:45 UTC   snap   -

Create a named boot environment

# Create a snapshot before making changes
debz-be create pre-nginx-upgrade

# Create with a date prefix automatically
debz-be create        # uses today's date: YYYY-MM-DD

Activate a boot environment

Activating a BE sets it as the target for the next reboot. The currently running system is not affected.

# Activate a snapshot for next boot (creates a clone internally)
debz-be activate pre-nginx-upgrade

# Reboot into it
systemctl reboot

Roll back to the previous boot environment

# Roll back to the most recent previous BE
debz-be rollback
Warning debz-be rollback rolls back the root dataset to the last snapshot. This is a destructive operation on the current root dataset — any changes made since the snapshot was taken will be lost. It does not affect /home, /var/log, or other data datasets.

ZFSBootMenu — interactive selection at boot

If you need to choose a different boot environment interactively — for example, after a failed upgrade that won’t SSH — hold a key at boot to pause at the ZFSBootMenu screen.

ZFSBootMenu’s keyboard shortcuts:

KeyAction
[Enter]Boot the selected environment
[e]Edit kernel command line for this boot
[k]Choose a kernel version for this environment
[s]Open a recovery shell (ZFSBootMenu shell, not the OS)
[r]Rollback the selected snapshot (dangerous)
[d]Set as default boot environment
[↑/↓]Navigate between environments
Note For VMs, ZFSBootMenu appears briefly and auto-boots. Send a break sequence or configure the VM to pause at boot. In Proxmox, set Boot Order timeout or use the noVNC console to interact with ZFSBootMenu during the brief window.

Recovery with debz-recovery

If the system is unbootable and you cannot use ZFSBootMenu (e.g. firmware issue, corrupted bootloader), boot the live ISO and use debz-recovery to import the pool and chroot in: 

# Boot live ISO, then:
debz-recovery

[debz-recovery] Scanning for ZFS pools...
[debz-recovery] Found: rpool on /dev/sdb, /dev/sdc
[debz-recovery] Importing rpool...
[debz-recovery] Mounting datasets to /target...
[debz-recovery] Chrooting into /target...

# Now inside the broken system's chroot
apt --fix-broken install
update-initramfs -u
exit

[debz-recovery] Unmounting /target...
[debz-recovery] Exporting rpool...
[debz-recovery] Done. Reboot the machine.

Boot environment naming conventions

Use consistent naming so snapshots are readable over time:

ScenarioRecommended name
Before routine upgradepre-upgrade-YYYY-MM-DD
Before major package changepre-nginx-4.0
Known good statestable-YYYY-MM-DD
Before config experimentpre-nftables-rewrite

Snapshot vs. clone boot environments

debz-be create creates a ZFS snapshot. Snapshots are read-only — ZFSBootMenu clones them on-demand when you select one for booting, creating a writable dataset. If you boot into a snapshot BE and it’s good, you can promote the clone to become the new primary using standard ZFS commands: 

# After booting into a cloned BE, promote it to primary
zfs promote rpool/ROOT/prod-node-01

# The clone is now the primary, original is now a clone
# Clean up the old primary if no longer needed
zfs destroy rpool/ROOT/prod-node-01-original
