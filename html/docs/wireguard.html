WireGuard Planes

debz uses four dedicated WireGuard interfaces, each with a fixed purpose. Separating planes means storage traffic, Kubernetes, management, and enrollment never share a path — and services bind to the right interface by design.

The four planes

Interface Network Port Purpose Services that bind here
wg0 10.77.0.0/16 51820 Enrollment only debz-autobootstrap listener
wg1 10.78.0.0/16 51821 Management Salt master, SSH, Web UI (:8080/:8081)
wg2 10.79.0.0/16 51822 Kubernetes kubelet, Cilium, kubeadm API server
wg3 10.80.0.0/16 51823 Storage NFS exports, iSCSI, ZFS replication, prometheus-node-exporter
Rule: never bind services to wg0. wg0 is enrollment-only and unreliable on some hardware. Salt, SSH, the Web UI, and all management tooling bind exclusively to wg1 (10.78.0.0/16). Services that accidentally bind to wg0 will appear to work on well-behaved hardware and silently fail on others.

Node IP addressing

Each node is assigned an integer index at stamp or enrollment time. The index maps directly to the last octet on every plane. The MASTER hub is always .1.

NodeIndexwg0wg1wg2wg3
master-01 (hub)110.77.0.110.78.0.110.79.0.110.80.0.1
kvm-01210.77.0.210.78.0.210.79.0.210.80.0.2
storage-01310.77.0.310.78.0.310.79.0.310.80.0.3
vdi-01410.77.0.410.78.0.410.79.0.410.80.0.4
node-NN10.77.0.N10.78.0.N10.79.0.N10.80.0.N

Formula: for plane wgX (X = 0..3), node at index N has IP 10.7(7+X).0.N.

Hub is always .1 on every plane

The MASTER node acts as the WireGuard hub. Its interface addresses are fixed and must never be reassigned. Nodes route all inter-plane traffic through the hub on wg0/wg1; wg2 and wg3 can use direct peer-to-peer routing between nodes once enrolled, but the hub still holds .1 on those networks to anchor Salt, DNS, and cluster control-plane endpoints.

Live peer updates — no restarts

New nodes are added to the hub without restarting any WireGuard interface. The hub’s debz-autobootstrap daemon calls wg set directly on the running interface: 

# Hub adds node-05 (index 5) to all four planes
wg set wg0 peer <node-05-pubkey> allowed-ips 10.77.0.5/32
wg set wg1 peer <node-05-pubkey> allowed-ips 10.78.0.5/32
wg set wg2 peer <node-05-pubkey> allowed-ips 10.79.0.5/32
wg set wg3 peer <node-05-pubkey> allowed-ips 10.80.0.5/32

# No wg-quick down/up needed — changes take effect immediately

Checking WireGuard status

# Show all interfaces and peer handshakes
wg show

# Check wg1 specifically — management plane
wg show wg1

# Verify interface addresses
ip addr show wg1

# Ping the hub on wg1 (management plane)
ping -c 3 10.78.0.1

# Check the peer table (all configured peers on wg1)
wg show wg1 peers

# See latest handshake times
wg show wg1 latest-handshakes

Setting up wg0 for enrollment

On a fresh node before enrollment, wg0 is the only pre-configured interface. It is configured with the node’s private key and the hub’s endpoint and public key:

# /etc/wireguard/wg0.conf on a new node (pre-enrollment)
[Interface]
Address = # assigned during stamp or DEBZ-SEED
PrivateKey = <node-private-key>
ListenPort = 51820

[Peer]
PublicKey = <hub-wg0-pubkey>
Endpoint = <hub-public-ip>:51820
AllowedIPs = 10.77.0.0/16
PersistentKeepalive = 25

# Bring up enrollment plane
wg-quick up wg0

# Verify connectivity to hub enrollment listener
ping -c 3 10.77.0.1

hub.env format

When a node successfully enrolls via wg0, the hub issues a signed hub.env file. This file contains everything the node needs to configure all four planes and join Salt. 

# hub.env — issued by debz-autobootstrap on the MASTER node
DEBZ_NODE_INDEX=5
DEBZ_HOSTNAME=kvm-05

# WireGuard keys and hub endpoints for all four planes
DEBZ_HUB_WG0_PUBKEY=<hub-wg0-pubkey>
DEBZ_HUB_WG1_PUBKEY=<hub-wg1-pubkey>
DEBZ_HUB_WG2_PUBKEY=<hub-wg2-pubkey>
DEBZ_HUB_WG3_PUBKEY=<hub-wg3-pubkey>
DEBZ_HUB_ENDPOINT=203.0.113.10   # hub public IP or DNS name

# Assigned IPs
DEBZ_WG0_IP=10.77.0.5
DEBZ_WG1_IP=10.78.0.5
DEBZ_WG2_IP=10.79.0.5
DEBZ_WG3_IP=10.80.0.5

# Salt master on wg1
DEBZ_SALT_MASTER=10.78.0.1

The node’s debz-autobootstrap client receives this file, writes WireGuard configs for wg1/wg2/wg3, brings all four interfaces up, and starts salt-minion pointed at 10.78.0.1.

wg0 reliability note wg0 uses port 51820. On some hardware (certain consumer NICs, some hypervisor vNIC drivers), this port’s packets are silently dropped or the handshake stalls. This is why wg0 is enrollment-only and not used for any operational traffic. If a node cannot enroll via wg0, manually copy hub.env to the node over any available path (USB, SSH to live session, serial console) and run debz-apply-role --hub-env /path/to/hub.env.

Debugging WireGuard

# Enable kernel debug logging (temporary, verbose)
echo module wireguard +p > /sys/kernel/debug/dynamic_debug/control

# Watch kernel messages for WireGuard events
dmesg -wH | grep -i wireguard

# Check if handshake is happening (non-zero = recently handshaked)
wg show wg0 latest-handshakes

# Verify UDP port is open on hub from node
nc -u -z <hub-public-ip> 51820 && echo "reachable"

# Check firewall is not blocking (on hub)
nft list ruleset | grep 5182

# Force a keepalive to re-trigger handshake
ping -c 1 10.77.0.1
