Proxmox + debz

debz runs well on Proxmox VE. This guide covers ISO upload, VM configuration, Secure Boot, golden image deployments, and standing up a full cluster on Proxmox.

Prerequisites

  • Proxmox VE 8.x or later
  • Enough storage for the VM disk — 40 GB minimum for server, 80 GB for desktop
  • UEFI + Secure Boot capable VM (OVMF firmware)

Upload the ISO

Upload the debz ISO to Proxmox storage via the web UI or scp:

# From a machine with the ISO, upload to Proxmox local storage
scp debz-trixie-amd64-server.iso root@proxmox-host:/var/lib/vz/template/iso/

# Verify it arrived
ssh root@proxmox-host "ls -lh /var/lib/vz/template/iso/"

Or use the Proxmox web UI: Datacenter → Node → local → ISO Images → Upload.

Create the VM

The critical VM settings for debz are UEFI firmware, an EFI disk, and Secure Boot initially disabled.

Via Proxmox web UI

  1. General: Name the VM, select a Node and VM ID
  2. OS: Select the debz ISO; Type = Linux, Version = 6.x - 2.6 Kernel
  3. System:
    • Machine: q35 (required for OVMF + PCIe)
    • BIOS: OVMF (UEFI)
    • Add EFI Disk: checked
    • Pre-Enroll keys: checked (loads Microsoft CA for shim)
    • Secure Boot: unchecked for initial install
  4. Disks: VirtIO SCSI, at least 40 GB for server / 80 GB for desktop
  5. CPU: at least 2 cores (4 recommended)
  6. Memory: at least 2048 MB (4096 recommended)
  7. Network: VirtIO (paravirtualised) on your chosen bridge

Via pvesh / qm CLI

# Create a server VM (100 = VMID, adjust as needed)
qm create 100 \
  --name debz-server-01 \
  --memory 4096 \
  --cores 4 \
  --cpu host \
  --machine q35 \
  --bios ovmf \
  --efidisk0 local-lvm:1,efitype=4m,pre-enrolled-keys=1 \
  --scsihw virtio-scsi-pci \
  --scsi0 local-lvm:40,discard=on \
  --ide2 local:iso/debz-trixie-amd64-server.iso,media=cdrom \
  --boot order=ide2 \
  --net0 virtio,bridge=vmbr0 \
  --agent enabled=1 \
  --ostype l26 \
  --tpmstate0 local-lvm:4,version=v2.0
Note The --tpmstate0 line adds a virtual TPM 2.0 device. This is optional — debz does not require TPM for basic Secure Boot operation (MOK-based signing works without TPM). Include it if you want TPM-backed key storage or future TPM integration.

VM_SECURE_BOOT and VM_TPM via deploy.sh

When using deploy.sh deploy, Secure Boot and TPM can be requested via environment variables. Both default to off; set to 1 to include them when the VM is created: 

# Create a Secure Boot + TPM VM via deploy.sh
./deploy.sh deploy \
  PROXMOX_HOST=proxmox.debz.local \
  VMID=101 \
  VM_NAME=debz-secure-01 \
  VM_SECURE_BOOT=1 \
  VM_TPM=1 \
  VM_DISK_GB=60

When VM_SECURE_BOOT=1, deploy.sh creates the EFI disk with pre-enrolled-keys=1 and sets the Secure Boot flag. The VM still boots for the first time with Secure Boot disabled so the installer can run; after the installer powers off, qm set flips the flag.

Running the install

Interactive (noVNC console)

  1. Start the VM: qm start 100
  2. Open the noVNC console in the Proxmox web UI
  3. The VM boots into the debz live session
  4. Run debz-installer --guided in the console or SSH to the live session’s IP
  5. Complete the 6-step installer

Unattended via SSH

# Start the VM
qm start 100

# Wait for the live session to boot and get an IP via DHCP
# Then copy the answer file and run the installer
scp answers.env root@<live-ip>:/tmp/
ssh root@<live-ip> "debz-installer --config /tmp/answers.env"

# The system powers off after install. Enable Secure Boot, then start:
qm set 100 --args "-global driver=cfi.pflash01,property=secure,value=on"
qm start 100

Enabling Secure Boot after install

After the installer powers the VM off, enable Secure Boot before the next boot. In Proxmox 8.x, Secure Boot state is controlled via the EFI disk settings:

  1. In the Proxmox web UI, go to the VM → Hardware → EFI Disk → Edit
  2. Check Secure Boot
  3. Start the VM
  4. In noVNC, MokManager appears — follow the Secure Boot + MOK enrollment steps
# Alternative: set via CLI (check your Proxmox version's exact flag)
qm set 100 --efidisk0 local-lvm:1,efitype=4m,pre-enrolled-keys=1,secure-boot=1

Golden images on Proxmox (deploy.sh golden)

For cluster deployments, build golden images once and stamp CoW clones for each node. The golden command builds a fully installed qcow2 using headless QEMU on the machine where deploy.sh runs. Once built, deploy.sh spawn uploads the clones to Proxmox and creates VMs from them. 

# 1. Build KVM golden image (runs headless QEMU locally)
./deploy.sh golden TEMPLATE=kvm GOLDEN_SIZE=60G

# 2. Stamp 4 clones (index 2..5, hostname kvm-02 through kvm-05)
./deploy.sh stamp TEMPLATE=kvm STAMP_COUNT=4 STAMP_BASE_HOSTNAME=kvm STAMP_BASE_INDEX=2

# 3. Spawn as Proxmox VMs (uploads qcow2, creates VMs)
./deploy.sh spawn TEMPLATE=kvm PROXMOX_HOST=proxmox.debz.local

# Each spawned VM boots and auto-enrolls via wg0 to the MASTER

See Golden Images for the full build pipeline and all environment variables.

ZFS topology testing with extra disks

To test mirror or raidz1 topologies in a Proxmox VM, add extra virtual disks at VM creation time. deploy.sh deploy supports this with VM_EXTRA_DISKS and VM_EXTRA_DISK_GB: 

# Create a VM with 3 data disks for raidz1 testing
./deploy.sh deploy \
  PROXMOX_HOST=proxmox.debz.local \
  VMID=103 \
  VM_NAME=debz-raidz1-test \
  VM_DISK_GB=40 \
  VM_EXTRA_DISKS=3 \
  VM_EXTRA_DISK_GB=40 \
  PROXMOX_DATA_STORE=local-zfs

# Inside the VM, use: DEBZ_ZFS_TOPOLOGY=raidz1
# DEBZ_ZFS_DATA_DISKS="/dev/sdb /dev/sdc /dev/sdd"
PROXMOX_DATA_STORE Set PROXMOX_DATA_STORE to your Proxmox storage pool name. Common values: local-lvm (default), local-zfs, ceph-pool. For ZFS-on-ZFS performance, set sync=disabled on the Proxmox host zvols after the VM is created.

VM disk on Proxmox ZFS storage

If your Proxmox host uses ZFS for storage, the VM disk is a zvol. debz inside the VM uses its own ZFS pools (rpool + bpool) on the virtual disk — this is ZFS-on-ZFS, which works fine but has some implications:

  • Disable synchronous writes on the zvol to avoid double-sync overhead: zfs set sync=disabled rpool/data/vm-100-disk-0 on the Proxmox host
  • Or use sync=standard inside the VM (default) and let Proxmox ZFS handle sync
  • Set the inner ZFS pool’s ashift to match the outer pool (usually ashift=12)
# On Proxmox host: disable sync on the zvol for VM performance
zfs set sync=disabled rpool/data/vm-100-disk-0

# The debz installer inside the VM sets ashift=12 by default

Snapshots and ZFS send from Proxmox

Proxmox offers built-in VM backup via its snapshot and vzdump mechanisms. For debz VMs, you have two levels of backup:

  • Proxmox level: vzdump creates a consistent snapshot of the entire VM disk (zvol snapshot on the host). This captures everything, regardless of the guest OS.
  • Guest level: debz’s own APT hooks and /srv timer run inside the VM, protecting the guest’s datasets against bad package installs.

Use Proxmox backup for disaster recovery and debz boot environments for operational rollback.

Networking: multiple debz VMs

For a cluster of debz VMs on Proxmox, a common pattern is:

# Create an isolated bridge for inter-VM traffic
# In Proxmox: Datacenter → Node → Network → Create → Linux Bridge
# Name: vmbr1, no gateway, no IP (L2 only)

# Wire up VMs: vmbr0 = uplink, vmbr1 = cluster mesh
qm set 100 --net1 virtio,bridge=vmbr1
qm set 101 --net1 virtio,bridge=vmbr1

# Inside the debz VMs, WireGuard planes are configured automatically
# during enrollment — WG encapsulates all cluster traffic over vmbr1
Tip For automated provisioning of many debz VMs from the same ISO, use deploy.sh golden + stamp + spawn. Each clone gets unique identity injected before first boot. No SSH or manual intervention needed per node — they enroll via wg0 automatically. See Cluster Bootstrap for the full enrollment flow.
