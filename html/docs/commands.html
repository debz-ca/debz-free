Command Reference

Every debz-* command on the live and installed system — organized by function, with usage examples.

Badge key: live-only runs on the ephemeral live desktop only  ·  CM only meaningful on a Cluster Manager node  ·  root requires root / sudo
User-facing Install & target Firstboot Cluster ops ZFS & system Network & security Live-only Systemd services

User-facing

debz-installer /usr/bin/
Wrapper that launches the TUI installer — detects gnome-terminal, falls back to xterm at 18pt dark.
Examples
# Launch from the live desktop terminal debz-installer
# Or click the installer icon on the GNOME dock # Same script — spawns a styled dark terminal window
debz-installer-tui /usr/bin/
The full bash/ncurses installer TUI — 7-step guided flow: disk, storage mode, hostname/profile, keyboard, timezone, network, user/password.
Examples
# Run interactively inside a terminal debz-installer-tui
# Non-interactive with a pre-filled answers file debz-installer-tui --config /mnt/usb/answers.env
debz-launch-installer /usr/bin/
Opens the installer in a dark terminal window. Called from the GNOME launcher icon — prefers gnome-terminal, falls back to xterm.
Examples
# Invoked by the GNOME dock icon, but also works from a terminal debz-launch-installer
debz-live-session-init /usr/bin/
Runs at live desktop login — sets wallpaper, dock, MOTD, and starts the autoinstall seed watcher.
Examples
# Called automatically by the GNOME autostart entry at login # To re-run manually (e.g. after editing dconf defaults): debz-live-session-init
debz-apply-live-wallpaper /usr/bin/
Sets the live session wallpaper via dconf.
Examples
# Reset to default debz wallpaper debz-apply-live-wallpaper # Override with a custom image DEBZ_WALLPAPER=/tmp/custom.jpg debz-apply-live-wallpaper
debz-deploy-tui CM /usr/local/sbin/
Cluster deploy TUI — spawn VMs, assign roles, monitor join progress across the fleet.
Examples
# Open the interactive cluster deploy terminal UI (run as root on CM) sudo debz-deploy-tui
# Use to bootstrap a 3-node etcd cluster + 2 workers interactively # Select nodes from the auto-discovered list, assign roles, confirm
debz-webui root /usr/local/bin/
Python HTTPS web UI server (port 8443) — nodes, ZFS, golden images, WireGuard, Ansible launcher, logs. Normally managed by systemd, but can be run directly for debugging.
Examples
# Check status (managed by systemd in production) systemctl status debz-webui # Run manually in the foreground for debugging sudo debz-webui
# Open in browser (self-signed cert — accept the warning) xdg-open https://localhost:8443

Install & target

debz-install-target root /usr/sbin/
Core installer engine — partitions disk, creates ZFS pools (bpool/rpool), runs debootstrap, writes all system config, installs bootloader. Accepts --config /path/to/answers.env for fully non-interactive installs.
Examples
# Interactive install — prompts for all values sudo debz-install-target
# Non-interactive with a seed file (CI / PXE / USB autoinstall) sudo debz-install-target --config /mnt/seed/answers.env
# Minimal answers.env example: DEBZ_DISK=/dev/sda DEBZ_STORAGE_MODE=zfs DEBZ_HOSTNAME=node01 DEBZ_USERNAME=admin DEBZ_PASSWORD=changeme DEBZ_PROFILE=server DEBZ_TIMEZONE=UTC
debz-stamp-identity root /usr/local/sbin/
Writes hostname, role, cluster name, and deployment intent (standalone / cluster-manager / join) into the installed system's /etc/debz/answers.env.
Examples
# Stamp a new node as a KVM host joining an existing cluster sudo debz-stamp-identity \ --hostname kvm01 \ --role kvm \ --infra-mode join \ --cluster-name prod
# Stamp a standalone ZFS storage node sudo debz-stamp-identity --hostname nas01 --role storage --infra-mode standalone
debz-autoinstall root /usr/local/sbin/
Watches for an answers.env file on removable media or kernel cmdline, then runs a fully non-interactive install. Used for USB seed-based provisioning.
Examples
# Place answers.env on a USB stick labelled DEBZSEED, boot debz ISO # debz-autoinstall.service fires automatically and installs without prompts
# Or pass the seed path via kernel cmdline: # debz.seed=http://192.168.1.10/answers.env # debz-autoinstall fetches it and calls debz-install-target --config
debz-apply-platform-holds root /usr/sbin/
Pins ZFS DKMS modules, kernel, and ZFSBootMenu from accidental apt upgrade removal.
Examples
# Apply holds after system install (done automatically by installer) sudo debz-apply-platform-holds # Verify held packages apt-mark showhold

Firstboot dispatchers

Each runs exactly once — gated by a /var/lib/debz/firstboot-*.done marker. debz-firstboot reads the node role and calls the appropriate specialization.

debz-firstboot
Dispatcher — reads DEBZ_INFRA_MODE and DEBZ_ROLE from /etc/debz/answers.env, calls the right firstboot script. On Free edition: configures Salt masterless + enables web UI. On Pro: full cluster dispatch.
Examples
# Managed by systemd — runs automatically on first real boot systemctl status debz-firstboot # Re-run manually (e.g. after editing answers.env to change role) sudo rm /var/lib/debz/firstboot-main.done sudo systemctl start debz-firstboot
debz-firstboot-master CM
Cluster Manager: generates admin SSH keypair, initialises state.db, brings up WireGuard wg1 (10.78.0.1/16), starts Salt master, creates the ansible service account + ED25519 keypair, writes hub.env, configures nginx to serve it, seeds Ansible inventory, starts reconciler and web UI.
Examples
# Called automatically by debz-firstboot when role=cluster-manager # To re-initialise a CM from scratch: sudo rm /var/lib/debz/firstboot-master.done sudo systemctl start debz-firstboot
# Check the resulting hub.env (node join config) cat /etc/debz/hub.env
debz-firstboot-kvm
KVM node: fetches hub.env from CM over plain LAN, derives WireGuard IP deterministically from LAN IP octets, installs Firecracker microVM runtime + jailer, configures libvirt ZFS storage pool, sets up nftables.
Examples
# Runs automatically when role=kvm — requires CM on LAN # Manual re-run after fixing a network issue: sudo rm /var/lib/debz/firstboot-kvm.done sudo systemctl start debz-firstboot
debz-firstboot-proxmox
Proxmox node: fetches hub.env, derives WG IP, installs Proxmox VE from bookworm repo with trixie apt pin (Priority: 900), creates API token, registers ZFS pool with pvesm, joins WireGuard mesh.
Examples
# Triggered automatically when role=proxmox # After completion, verify Proxmox is up: systemctl status pveproxy pvesm status
debz-firstboot-etcd
etcd member: fetches join token from CM, installs etcd, joins the cluster over wg1.
Examples
# Runs when role=etcd — verify cluster health after completion: ETCDCTL_API=3 etcdctl member list --endpoints=https://10.78.0.1:2379
debz-firstboot-worker
Kubernetes worker: installs containerd, joins k8s cluster via kubeadm join over wg2.
Examples
# Runs when role=worker — after completion check node joined: kubectl get nodes # run from CM or any node with kubeconfig
# Fetch a fresh join token and run manually if firstboot failed: sudo debz-k8s-join
debz-firstboot-lb
Load balancer: installs HAProxy, configures it for control plane VIP on wg1.
Examples
# Runs when role=lb — check HAProxy after completion: systemctl status haproxy haproxy -c -f /etc/haproxy/haproxy.cfg
debz-firstboot-monitoring
Monitoring node: installs Prometheus, Grafana, and Loki; scrapes all wg1 nodes.
Examples
# Runs when role=monitoring — open Grafana after completion: xdg-open https://10.78.0.<node>:3000
# Check Prometheus targets are all UP: curl -s http://localhost:9090/api/v1/targets | jq '.data.activeTargets[].health'
debz-firstboot-firewall
Dedicated firewall/gateway node: configures nftables + WireGuard exit plane (wg4).
Examples
# Verify nftables rules loaded after firstboot: sudo nft list ruleset | head -40
debz-firstboot-landing
Bastion / jump-host node: SSH proxy config, access logging, two-factor setup.
Examples
# Proxy through the landing node to any cluster node: ssh -J admin@landing.cluster.local admin@10.78.0.5

Cluster operations

debz-reconcile CM /usr/local/sbin/
Python daemon — polls Salt grains every 30 s, updates state.db (nodes, services, events), auto-generates Ansible inventory at /etc/debz/inventory and mirrors it to /srv/ansible/inventory/hosts.ini.
Examples
# Check reconciler status systemctl status debz-reconcile # View live logs journalctl -fu debz-reconcile
# Inspect the auto-generated Ansible inventory cat /srv/ansible/inventory/hosts.ini
debz-node-register CM /usr/local/sbin/
Manually register a node into state.db — useful for bare-metal nodes that can't self-register.
Examples
# Register a bare-metal storage node manually sudo debz-node-register \ --hostname nas01 \ --ip 192.168.1.42 \ --wg-ip 10.78.0.42 \ --role storage
# List all registered nodes sqlite3 /var/lib/debz/state.db "SELECT hostname, role, status FROM nodes;"
debz-autobootstrap CM /usr/local/sbin/
Scans the LAN for new debz nodes, registers them in state.db, and pushes their first role via Salt/Ansible.
Examples
# Run a manual LAN scan from the CM sudo debz-autobootstrap --scan # Run with verbose output to see discovery details sudo debz-autobootstrap --scan --verbose
debz-spawn root /usr/local/sbin/
Spin up a new VM from a golden image — creates a CoW ZFS clone, stamps identity, and boots it on the target hypervisor (KVM or Proxmox).
Examples
# Spawn a new worker node from the base golden image sudo debz-spawn \ --image debz-base-2025-03 \ --hostname worker04 \ --role worker \ --host kvm01
# Spawn a Proxmox VM with custom RAM/CPU sudo debz-spawn --image debz-base --hostname pve-node02 --host proxmox01 --mem 8G --cpus 4
debz-replicate root /usr/local/sbin/
Clone a running node to another host via ZFS send/recv over wg3 (replication plane).
Examples
# Replicate worker03's root dataset to a standby host sudo debz-replicate --source worker03 --dest standby01 --pool rpool/ROOT/worker03
# Full incremental sync (only changed blocks since last snap) sudo debz-replicate --source nas01 --dest nas-dr --incremental
debz-apply-role root /usr/local/sbin/
Push a Salt state or Ansible playbook to a node or node group — wraps salt-call or ansible-playbook with the correct inventory and key.
Examples
# Apply the storage role to a single node via Salt sudo debz-apply-role --node nas01 --role storage # Apply to all nodes tagged as workers sudo debz-apply-role --group workers --role k8s_worker
# Run a custom Ansible playbook with the managed inventory sudo debz-apply-role --playbook /srv/ansible/playbooks/harden.yml --limit nas01
debz-golden root /usr/local/sbin/
Capture a golden image snapshot from a running node — freezes state, creates a ZFS send stream, and registers it in the golden image library.
Examples
# Capture a golden image of worker01's current state sudo debz-golden --source worker01 --name debz-worker-2025-03-18
# List all available golden images sudo debz-golden --list # Delete an old image sudo debz-golden --delete debz-worker-2025-01-05
debz-audit CM /usr/local/sbin/
Query and display the audit trail from state.db — node events, role changes, key rotations, installs.
Examples
# Show all events in the last 24 hours sudo debz-audit --since 24h # Filter to a specific node sudo debz-audit --node worker02
# Export to JSON for SIEM ingestion sudo debz-audit --since 7d --format json > /tmp/audit-week.json
debz-poof root /usr/local/sbin/
Cleanly destroy a node — stops services, removes from state.db, revokes WireGuard peer, optionally wipes disk.
Examples
# Decommission worker04 cleanly (keeps disk intact) sudo debz-poof --node worker04
# Destroy and wipe disk (irreversible — use with care) sudo debz-poof --node worker04 --wipe # Dry run to see what would happen sudo debz-poof --node worker04 --dry-run

ZFS & system

debz-be root /usr/local/sbin/
Boot environment manager — create, list, activate, clone, and destroy ZFS boot environments. Wraps ZFSBootMenu's zbm-promote/zbm-clone with a friendly interface.
Examples
# List all boot environments sudo debz-be list # Create a new BE before a risky upgrade sudo debz-be create pre-upgrade-2025-03-18
# Activate an older BE (takes effect on next boot) sudo debz-be activate pre-upgrade-2025-03-18 # Clone a BE for testing sudo debz-be clone stable testing-new-kernel
debz-snapshot root /usr/local/sbin/
Take a manual ZFS snapshot across all rpool datasets. Also run automatically by the 15-minute systemd timer before any apt operation.
Examples
# Take a manual snapshot with an optional tag sudo debz-snapshot pre-config-change # List recent snapshots zfs list -t snapshot -s creation -o name,creation | tail -20
# Roll back to a specific snapshot sudo zfs rollback rpool/ROOT/hostname@pre-config-change
debz-recovery root /usr/sbin/
Drop into a guided recovery shell — imports pools, mounts datasets, offers rollback and chroot options. Invoked from ZFSBootMenu or a rescue USB.
Examples
# Run from a debz live USB after booting into rescue mode sudo debz-recovery # Directly import and chroot into the installed system sudo debz-recovery --chroot
# Import pool and roll back to a previous snapshot non-interactively sudo debz-recovery --rollback rpool/ROOT/hostname@pre-upgrade
debz-upgrade root /usr/local/sbin/
Managed apt upgrade with automatic pre-upgrade ZFS snapshot and rollback instructions on failure. Respects package holds set by debz-apply-platform-holds.
Examples
# Safe upgrade — snapshots first, then upgrades sudo debz-upgrade
# Upgrade and automatically clean old apt cache sudo debz-upgrade --clean # Dry run — show what would be upgraded sudo debz-upgrade --dry-run

Network & security

debz-dns-api CM /usr/local/sbin/
Internal cluster DNS REST API — SQLite-backed, serves A/AAAA records for all wg1 nodes. Runs as a systemd service on port 5353.
Examples
# Look up a node's wg1 IP via the internal DNS API curl -s http://10.78.0.1:5353/resolve?name=worker02 | jq . # List all registered names curl -s http://10.78.0.1:5353/records | jq .
debz-dns-register /usr/local/sbin/
Register a node's WireGuard IPs into the internal DNS — called automatically at firstboot and after WG key rotation.
Examples
# Re-register this node after a WG key rotation sudo debz-dns-register
# Register a different hostname/IP explicitly sudo debz-dns-register --hostname nas01 --wg-ip 10.78.0.42
debz-issue-cert CM /usr/local/sbin/
Issue a TLS certificate from the cluster CA — SAN includes hostname, wg1 IP, and LAN IP.
Examples
# Issue a cert for a new service node sudo debz-issue-cert --hostname monitor01 --lan-ip 192.168.1.50 --wg-ip 10.78.0.50
# Issue a cert for a custom service SAN sudo debz-issue-cert --hostname grafana.cluster.local --extra-san metrics.cluster.local
debz-tls-cert /usr/local/sbin/
TLS certificate helper — sign CSRs, check expiry, renew with CM CA.
Examples
# Check expiry of the current node cert sudo debz-tls-cert check # Renew the cert (fetches new one from CM CA) sudo debz-tls-cert renew
debz-clevis-seal root /usr/local/sbin/
Seal the ZFS encryption key to the TPM using Clevis — enables unattended encrypted boots without passphrase prompt.
Examples
# Seal the ZFS passphrase to the local TPM2 chip sudo debz-clevis-seal --pool rpool --tpm2
# Seal to both TPM2 and a Tang server for redundancy sudo debz-clevis-seal --pool rpool --tpm2 --tang https://tang.cluster.local
# Test that the seal works (dry run, no passphrase prompt) sudo debz-clevis-seal --test --pool rpool
debz-k8s-join /usr/local/sbin/
Fetch a fresh kubeadm join token from the CM over wg1 and execute it — called by debz-firstboot-worker.
Examples
# Re-join a node that lost cluster membership sudo debz-k8s-join
# Print the join command without executing (for manual inspection) sudo debz-k8s-join --print-only

Live-only

These run on the ephemeral live desktop. They do not persist after reboot.

debz-live-ssh-init live-only /usr/local/sbin/
Generates a fresh ED25519 keypair at live boot, writes the public key to /run/debz-live-pubkey, and displays it in the MOTD. No hardcoded keys — every live session is unique.
Examples
# View the current live session's public key cat /run/debz-live-pubkey # Copy it into your local authorized_keys to SSH into the live session ssh-copy-id -i /run/debz-live-pubkey live@<live-ip>
# Check when the key was generated and its fingerprint ssh-keygen -l -f /run/debz-live-pubkey
debz-live-log-ship live-only /usr/local/sbin/
Periodically ships live session logs to the Cluster Manager if one is detected on the LAN — useful for fleet deployment monitoring.
Examples
# Check if the log shipper found a CM on the LAN journalctl -u debz-live-log-ship --no-pager -n 20
# Run a manual ship to a specific CM IP sudo debz-live-log-ship --cm 192.168.1.10
debz-live-mok-enroll live-only /usr/local/sbin/
Enrolls the debz MOK signing key for Secure Boot in the live session — required before ZFS DKMS modules can load on Secure Boot systems.
Examples
# Enroll the debz MOK key (prompts for a one-time password, reboot to confirm) sudo debz-live-mok-enroll
# After reboot and MOK enrollment, verify ZFS loaded: modinfo zfs | grep filename

Systemd services

These run as systemd units — managed via systemctl. Listed here for quick reference when reading logs or checking status.

debz-firstboot.service
One-shot: runs the debz-firstboot dispatcher on first real boot after install.
Examples
systemctl status debz-firstboot journalctl -u debz-firstboot --no-pager
debz-webui.service
Keeps the Python web UI running on port 8443. Restarts automatically on failure.
Examples
systemctl restart debz-webui journalctl -fu debz-webui # follow logs
debz-reconcile.service
Persistent daemon on the CM: Salt grain poll → state.db → Ansible inventory, every 30 s.
Examples
systemctl status debz-reconcile journalctl -fu debz-reconcile
debz-apt-mirror.service
Serves the embedded darksite APT pool at http://127.0.0.1:3142/apt for offline package installs.
Examples
# Verify the darksite mirror is serving curl -s http://127.0.0.1:3142/apt/dists/trixie/Release | head -5
debz-snapshot.service / .timer
15-minute recurring ZFS snapshot of all rpool datasets.
Examples
systemctl list-timers debz-snapshot.timer journalctl -u debz-snapshot --since today
debz-apt-snapshot.service
Triggered by the apt pre-invoke hook — snapshots all rpool datasets before every package install or upgrade.
Examples
# Confirm the apt hook is registered cat /etc/apt/apt.conf.d/99debz-snapshot # Check the last apt-triggered snapshot zfs list -t snapshot -s creation | grep apt | tail -5
debz-autobootstrap.service / .timer
Periodic LAN scan for new debz nodes to auto-register. Runs every 5 minutes on the CM.
Examples
systemctl list-timers debz-autobootstrap.timer journalctl -u debz-autobootstrap --since -1h
debz-dns-api.service
Internal cluster DNS API on port 5353 — runs on the CM.
Examples
systemctl status debz-dns-api curl -s http://10.78.0.1:5353/records | jq .
debz-live-ssh-init.service
Live-only: generates a fresh ED25519 SSH keypair at live boot.
Examples
systemctl status debz-live-ssh-init cat /run/debz-live-pubkey

Helper scripts

debz-promtail-setup /usr/local/sbin/
Configures Promtail for shipping logs to a Loki instance on the monitoring node over wg1.
Examples
# Configure Promtail pointing at the monitoring node sudo debz-promtail-setup --loki-url http://10.78.0.50:3100
# Re-run after the monitoring node's wg1 IP changes sudo debz-promtail-setup --loki-url http://monitor01.cluster.local:3100 systemctl restart promtail
